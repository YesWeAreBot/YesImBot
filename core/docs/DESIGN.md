# Athena Core 设计方案（会话与响应控制流）

## 1. 文档定位

- 本文档定义 `core` 在“会话与响应控制流”方向的目标能力契约，仅描述完成后应达到的功能状态。
- 本文档面向核心维护者，用于评审范围边界、阶段目标和验收标准。
- 本文档不包含代码层实现细节（框架 API 选型、类结构、调度算法），但包含对外可观测契约（目录语义、配置语义、生命周期边界）。

## 2. 设计范围

- 本轮范围聚焦单频道主代理（main agent）的完整响应生命周期：事件接入、会话状态演进、模型回合控制、工具回合衔接、超时与恢复。
- 本轮纳入工作区与配置分层能力，以及插件系统的对外扩展定位。
- 旧版分支中的改进意见仅保留与当前 `core` 方向一致的能力语义。

### 2.1 术语定义

- 有效输入事件：通过平台接入且通过基础过滤规则（非机器人自身消息、结构完整、未被去重规则丢弃）的频道事件。
- 意愿判断：系统基于规则信号与模型信号决定“是否触发响应”的决策过程，输出应至少包含“触发/不触发 + 原因类别”。
- 高风险动作：可能导致越界访问、外部副作用扩大或不可逆影响的文件与执行类动作。
- 会话结束态：一次响应生命周期的最终状态，限定为正常结束、主动中止、超时终止、错误终止。

## 3. 终态目标

### 3.1 会话与响应主链路

- 事件即会话事实：频道内有效输入事件均进入对应 agent session，响应是否触发不影响事件入链。
- 会话即响应边界：每个频道具备独立会话状态和独立响应状态，并发输入不破坏单频道顺序一致性。
- 中断后可继续：响应在超时、取消或故障后保留继续语义，后续触发可在同一会话目标下恢复推进。
- 响应可收敛：每次响应都能收敛到明确结束态（正常结束、主动中止、超时终止、错误终止）。

### 3.2 工具回合与消息语义

- 工具调用采用原生 `tool_call` 语义，文本输出与工具调用统一纳入响应生命周期。
- 模型文本与工具结果形成可验证闭环，支持“计划-执行-反馈-继续/结束”的一致语义。
- 输出到频道的消息与会话中记录的模型输出保持可对应关系。
- 内部推理内容不进入对外消息流，避免污染用户可见输出。
- 所有外发消息默认仅面向当前频道，不通过跨频道发信工具绕过会话边界。

### 3.3 上下文连续性

- 长会话下支持上下文可控退化，结合裁剪与摘要维持连续对话能力。
- 退化后仍保留关键任务线索、最近交互事实与恢复所需最小上下文。

### 3.4 能力固定性约束

- 不做动态 capability 发现，不做运行时动态工具发现。
- agent 在 session 初始化时即确定当前 IM 平台可用能力集合与工具集合。
- 同一 session 生命周期内，能力视图保持稳定，不发生不可解释漂移。
- 运行中插件新增或卸载仅影响后续新建 session，不影响当前 session 的冻结能力集合。

## 4. 工作区与配置分层目标

### 4.1 工作区隔离

- 每个频道对应一个持久工作区，频道内 session 共享该工作区状态。
- 默认按 `basePath/<channel>/` 进行频道级隔离。
- 频道之间不共享工作目录状态。

### 4.2 目录契约

完成态下工作区满足以下目录语义：

```text
<basePath>/<channel>/
  settings.json
  models.json
  session/
    <time>_<uuid>.jsonl
  workspace/
    SOUL.md
    AGENTS.md
    MEMORY.md
```

- 上述目录为默认契约示例，允许等价实现；验收以可观测行为一致性为准。

### 4.3 配置分层

- 当工作区配置不存在时，系统使用全局配置。
- 当工作区配置存在时，与全局配置进行深度合并后生效。
- 合并后配置在单 session 生命周期内应保持稳定语义。

### 4.4 文件与执行安全边界

- 默认拦截工作区外文件访问。
- 默认启用执行沙盒约束，避免危险操作直接落地。
- 越界或高风险动作的处理行为必须确定、可观测、可审计。

## 5. 插件系统目标（对外扩展点）

- `core` 保留插件系统作为长期对外扩展能力，不将工具、skill、hook 全量硬编码在核心主链路。
- 扩展契约以 `packages/plugin-sdk` 为主，支持注册工具、skill、hook。
- 插件能力的可发现性与可用性以 session 初始化时冻结的能力集合为准；运行中插件变更仅对后续新建 session 生效。
- 插件必须遵守工作区边界与安全约束，不得绕过文件或执行防护。
- 插件异常不应破坏会话状态一致性，系统需具备可判定的降级或终止行为。

## 6. 阶段目标（风险递进）

### V1：主链路稳定

- 完成“事件接入 -> 会话入链 -> 触发判断 -> 单次响应 -> 结束判定”闭环。
- 在高频输入场景下保证不丢事件、不乱序、不重复结束。
- 响应中断后可在后续触发中继续，不退化为全新对话语义。
- 支持基础工具回合闭环：文本输出与工具反馈在同一响应生命周期中收敛。
- 阶段验收门槛：单次响应闭环成立，且“响应中新增事件”可被保留并在后续回合推进。

### V2：多步回合协同

- 支持复杂多步响应中“文本与工具交替”的稳定状态推进。
- 支持响应进行中接收新事件并正确衔接后续回合。
- 触发策略从静态规则升级为“规则 + 意愿判断”协同，减少误触发与漏触发。
- 阶段验收门槛（通过）：多步回合的每一步均可判定结束态；响应中新增事件在后续回合可见并被消费；工具异常后会话仍可进入合法结束态。

### V3：鲁棒性与退化质量

- 在长会话、工具失败、反复中断等压力场景下保持输出稳定与上下文连续。
- 上下文压缩后仍保留关键任务信息和可恢复能力。
- 为后续多 agent 协同（main/heartbeat/knowledge）提供稳定接口前提。
- 阶段验收门槛（通过）：在长会话与故障注入回归用例中，系统仍可进入合法结束态，且关键任务目标、最近用户指令与未完成事项可恢复。

## 7. 非目标

- 不定义具体实现方式、框架 API 组合或代码组织细节。
- 不定义跨频道协同协议（A2A/事件总线）与共享记忆协议。
- 不展开 heartbeat/knowledge agent 的调度拓扑、资源策略与执行细节。
- 不覆盖供应链安全、主机加固等全量安全治理体系。
- 不定义插件生态层机制（市场、分发、签名托管等）。
- 不承诺兼容旧版 horizon/capability/timeline 的行为级实现。

## 8. 统一验收标准（跨阶段）

- 状态可判定：任一频道会话可查询到当前状态、最近结束态类型、是否存在待续回合。
- 事件不丢失：对任意测试窗口内的有效输入事件，会话记录可完成一一对账；响应中新增事件在后续回合可见。
- 响应可收敛：每次响应必须落入定义的会话结束态枚举，并记录对应原因类别。
- 上下文连续：在“长会话 + 压缩”回归用例中，关键任务目标、最近用户指令和未完成事项可被正确恢复。
- 边界可执行：越界文件访问和受限执行动作必须被阻断，并产生日志审计字段（频道、会话、动作类型、判定结果）。
- 扩展可控：插件异常场景下主链路仍可终止到合法结束态；恢复后可继续处理后续事件。

## 附录 A：旧版碎片思路到当前目标映射

- 旧：按频道隔离 session、即时入链、响应可继续 -> 新：事件即会话事实 + 会话即响应边界 + 中断后可继续。
- 旧：移除 horizon/event manager/timeline 中介编排 -> 新：响应状态以内聚会话语义为唯一事实来源。
- 旧：工具执行从 JSON 文本解析迁移 -> 新：统一原生 `tool_call` 回合闭环。
- 旧：响应中出现新消息时后续衔接 -> 新：并发输入下事件不丢失、状态可解释、响应可收敛。
- 旧：超时后中断并等待下一次触发 -> 新：中断后保持继续语义，不重建会话。
- 旧：软硬裁剪 + summary -> 新：上下文可控退化且保持任务连续性。
- 旧：AGENTS/SOUL/MEMORY 记忆注入 -> 新：工作区目录契约与配置分层能力。
- 旧：capability 驱动（已废弃） -> 新：移除动态 capability/工具发现，改为初始化固定能力集合。
- 旧：跨频道共享与协同 -> 新：不纳入本阶段目标，作为后续演进议题。
