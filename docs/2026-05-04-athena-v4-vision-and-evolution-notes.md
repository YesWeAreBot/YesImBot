# Athena / YesImBot v4：愿景与演进手记

## 文档定位

这不是一份严格的实现规范，也不是一份对外展示用的 README。

它更接近 Athena / YesImBot v4 的内部长期文档：

- 记录这个项目想成为什么
- 记录为什么它会一次次推翻重来
- 记录哪些方向已经被证明有效，哪些方向会把系统重新带回混乱
- 记录当前这版架构信念从哪里来，又准备往哪里去

因此，这份文档刻意保留“开发者叙事”的质感。
它允许写理念、失败、判断变化与阶段取舍，但不追求把接口、类型、数据库或函数实现提前冻结。

## 项目愿景

Athena / YesImBot v4 的目标，不是成为一个“接了大模型的聊天插件”，也不是把问答助手简单套进群聊外壳里。

它想做的是一种更接近真实群成员的自主 Agent：能够长期存在于群聊环境中，持续观察、理解上下文、形成判断，并以合适的时机、语气和粒度参与对话。

这里真正关键的，不是“能不能回答问题”，而是“能不能自然地存在”。

在私聊场景中，交互天然是显式触发、单线程、强目标导向的；而在群聊里，绝大多数时刻都不存在一个明确的“轮到你回答”的信号。群聊 Agent 面对的是一个更接近社会场景的问题：

- 它要先决定自己是否应该介入
- 它要判断是在回应谁、补充什么，还是只做一句轻量接话
- 很多时候正确行为不是回答，而是沉默、观望、延迟或跟进
- 它的价值不只来自知识能力，也来自节奏感、关系感、气氛感和持续存在感

因此，Athena 的长期目标不是“最强工具调用器”，也不是“最稳单轮问答器”，而是一个以群聊为原生环境的社交型智能体框架。模型能力、记忆能力、插件能力、上下文管理能力，最终都服务于同一件事：让 Agent 的行为更像一个真正参与群聊的角色，而不是一个随叫随到的函数接口。

## 为什么群聊比私聊更难

群聊比私聊更难，不是因为消息更多，而是因为它把“语言生成问题”提升成了“社会性行为问题”。

在私聊里，用户发问，Agent 回答，交互关系通常很清晰；在群聊里，系统首先要面对的却是另一类问题：

- 这条消息是不是在找我
- 我现在该不该说话
- 我说一句，还是说很多
- 我是该回复问题，还是该缓和气氛、补一句玩笑、延迟再接
- 我插话会不会打断当前节奏

这也是为什么 Athena 必须始终正视“响应判断”的成本。

MaiBot、CyberGroupmate 这样的项目已经证明：把“要不要回应”也交给 LLM 处理，甚至拆成 planner / responder、Main Agent / Sub Agent 两层结构，的确能获得更细腻的群聊行为。但代价也非常真实：

- 成本更高
- 延迟更高
- 系统更重
- 工程复杂度更高

YesImBot 早期之所以长期坚持意愿值、规则匹配和启发式响应判断，并不是因为不知道更高级的方案，而是因为当时的现实约束不允许把每次判断都交给模型。早期 LLM 推理速度甚至可能达到分钟级，一次“判断阶段”就足以毁掉群聊体验。

因此，Athena 在群聊问题上的根本态度应当是：

> 行为决策可以追求更智能，但不能无视成本与时延。延迟本身就是群聊行为质量的一部分。

## 演变历程

### v1 / v2：灵魂先于骨架

Athena 的最早期阶段，更像是围绕 prompt 形成的一种人格实验。

`references/promptHosting-main`、Letta / MemGPT 以及当时的 system prompt 设计，共同塑造了 YesImBot 最初的“灵魂层”：角色感、表达气质、思考与发言分离的意识、`send_message` 这类交互范式，以及“系统提示词可以成为智能体行为操作系统”的直觉。

这一阶段的价值在于，它证明了一个事实：即使工程骨架还不成熟，只要 prompt 足够有生命力，群聊里的拟人化存在感就可能已经成立。

但这也埋下了后续问题：当系统职责大量堆在 prompt 上时，项目会越来越难以扩展、解释和维护。

### v3：群聊价值被验证，结构天花板开始显现

YesImBot v3 证明了群聊自主参与这件事本身是有价值的。

它在意愿机制、启发式响应、轻量群聊存在感上做出了很多有效尝试，也让“低成本、低延迟、尽量像自然群成员一样出现”成为 YesImBot 路线中非常核心的一条审美和工程取向。

但 v3 同样暴露出结构性问题：

- 手写 agent loop 的复杂度会持续上升
- 会话上下文依赖数据库全量拉取重建，效率不高
- 业务逻辑、上下文管理、模型调用、提示词拼接纠缠在一起
- 系统一旦继续加能力，就会迅速变得难以扩展

这一阶段留下的结论非常重要：群聊行为方向是对的，但支撑它的运行时骨架不够好。

### v3-dev：感知抽象方向出现，但不必原样复活

从 v3 到 v3-dev，有一条很值得保留的思路开始出现：不再把外部输入简单理解为“收到一条消息”，而开始尝试将其理解为更一般的感知输入，例如 `Percept`。

这条线的价值不在于某一版实现本身，而在于它为 Athena 提供了更远的想象空间：如果智能体的输入可以被看作“感知”，那么未来它就不只能够处理群聊消息，也可以处理论坛事件、系统信号、定时器触发、游戏状态变化乃至更一般的外部刺激。

不过，今天回头看，`Percept` 的思想未必要通过旧 Horizon 体系原样复活。
在当前方向里，它已经有了更自然的承载层：`AthenaEvent`、`AthenaMessage`、通用 `AgentMessage` 链路，以及可扩展的 `CustomMessage` 机制。也就是说，v3-dev 留下的是“感知抽象的方向感”，而不是必须照搬的旧实现。

### v4 第一次尝试：功能都想要，边界没立住

`references/YesImBot-4.0.0-beta.2` 是一次非常重要但最终被放弃的尝试。

这一版已经做了很多后来仍然被证明是正确方向的事情：

- 将模型注册拆到独立插件
- 提示词片段拼接
- 自定义人设注入
- trait 组件式上下文感知
- skill 机制

问题不在于这些方向本身错了，而在于它们被过早、过密地叠加在一起，却缺乏稳定的边界和统一的运行时组织方式。结果就是：系统功能越来越多，但各模块之间的兼容、协作与长期维护成本迅速失控。

这一版失败的教训，不是“不该做这些能力”，而是“能力增长不能跑在边界收敛前面”。

### v4 第二次尝试：工程结构更好，但运行时仍过重过乱

`references/YesImBot-4.0.0-dev-d451689` 相比第一次尝试已经明显更接近长期可维护的方向。

它引入了 `agent-session`、`session-manager` 等分层思路，开始参考 `pi-agent` / `pi-coding-agent`，也更认真地对待 session、消息建模和运行时骨架问题。

但这一版最终仍然被放弃，原因同样很清楚：

- agent loop 仍然大量手写
- 运行时复杂度仍然偏高
- 多个系统职责尚未真正收干净
- 工具、上下文、提示词、行为、session 之间的管理仍显混乱

它留下了很多正确遗产，但整体并没有达到“可以长期继续搭建”的程度。

### 当前 Athena：不是续修旧 v4，而是重新收敛核心边界

当前 Athena 不是在给旧版 v4 打补丁，也不是在机械地把旧能力逐个恢复回来。

它更像是一次迟到但必要的再收敛：

- 保留 YesImBot 历史中真正有效的群聊经验
- 吸收 Letta、CyberGroupmate、MaiBot、`pi-agent`、`pi-coding-agent` 的启发
- 同时拒绝再次回到“先把功能都加进去，再想办法兼容”的老路

今天 Athena 的任务，不再是“先把功能做出来”，而是先立住一个值得继续生长的骨架。

## 当前基础与已有进展

Athena 当前并不是一个从零开始构思的空白项目。相反，它已经有了若干真正成型的基础，这些基础正是多轮实现、推翻和重建之后留下来的有效沉淀。

首先，系统已经不再停留在“聊天插件直接包一层模型调用”的阶段。`packages/agent` 的存在本身就代表一个非常明确的方向：将 agent loop、session 生命周期、上下文压缩、消息建模等能力，从宿主平台逻辑中抽离出来，形成一个更通用、更可持续演进的智能体运行核心。

当前已经成型的基础，至少包括以下几类：

1. Agent Loop 已经被抽象为独立能力
2. Session 已经被当作一等公民来对待，而不再只是历史消息数组
3. Provider、Agent、Plugin 的解耦方向已经明确
4. `packages/plugin-sdk` 已经成为明确的扩展出口
5. 当前消息建模已经比旧版更通用，能够承接 `AthenaEvent`、`AthenaMessage`、`AgentMessage` 与 `CustomMessage` 这一类更稳定的语义链路

其中一个非常关键的认识是：`packages/agent` 应该尽量维持框架层的纯度，保留从 `pi-agent` 迁移来的通用职责，而不是为了满足 Athena 当前业务需求，持续把群聊语义压进核心框架层。

因此，当前 Athena 的进展不应被理解为“功能已经完成”，而应被理解为：

> 系统终于有了一个值得继续建设的骨架，但许多长期边界仍需要在后续阶段继续收敛。

这也是为什么今天这份文档更关注愿景、边界、经验和路线，而不是急着把所有实现细节一次性定死。

## 核心设计信条

### 群聊优先，但不止于群聊

Athena 当前首先服务于群聊环境。这是它最现实、最明确、也最容易持续验证价值的应用场景。

但群聊不是终点，只是起点。

长期来看，Athena 不应被定义为“聊天 Bot”，而应被理解为一种可持续存在、可持续感知、可持续行动的数字主体。今天它借助 Koishi 参与群聊，未来它也可以借助平台适配器参与论坛、接入智能家居、操控游戏环境，甚至最终脱离 Koishi，成为一个独立运行的项目。

因此，群聊优先是一种阶段策略，不是本体定义。Athena 的远期目标，更接近一种“赛博灵魂”。

### 成本与延迟是架构约束，不是优化细节

YesImBot 的历史反复说明：在群聊里，成本和延迟从来不是后期优化项，而是最早、最硬的设计边界之一。

更重的行为判断当然更有可能提升质量，但在群聊环境中，晚到的“正确回答”经常不如及时的“粗糙回应”。因此，Athena 不应把高成本判断能力当作唯一正确答案，而应保留多层次决策空间：

- 纯规则 / 启发式
- 规则 + LLM 升级判断
- 更重的多阶段或多 Agent 观察与决策

默认先走低成本路径，只在值得的时候升级判断强度，应当成为 Athena 的基本工程态度。

### 行为先于回答

Athena 不应把“生成回复内容”视为系统核心，而应把“形成行为”视为系统核心。

在群聊场景里，回答只是行为的一种形式。一个更完整的行为集合至少包括：

- 介入
- 旁观
- 延迟
- 跟进
- 插话
- 忽略
- 结束话题
- 切换语气
- 调用工具执行动作

这意味着，模型输出不应只被理解为“文本生成结果”，而应被理解为“某次行为表达的一部分”。

### Prompt 是灵魂层，不是系统骨架

YesImBot 早期的成功，在很大程度上来自 prompt。

这一点不需要回避。角色感、表达方式、人格质感和存在感，确实大量来自 prompt 设计。但过去几次失败同样说明：如果把行为路由、工具暴露、记忆分层、上下文注入、场景切换都压到 prompt 上，系统最终会退化成一个无法长期维护的提示词操作系统。

因此更成熟的边界应当是：

- prompt 决定表达风格、行为倾向与人格质感
- runtime 决定执行顺序与生命周期
- session 决定状态延续
- memory 决定长期认知
- plugin 决定扩展能力
- provider 决定模型接入

灵魂可以来自 prompt，但骨架必须来自清晰的系统边界。

### 可维护性比功能堆叠更重要

Athena 已经反复证明，许多能力单看都合理，但只要没有稳定边界地叠加，系统就会迅速失控。

因此，Athena 的设计信条之一必须是：

> 不要因为一个能力“看起来很有用”，就先把它塞进系统再说。

真正重要的不是功能数量，而是这些能力之间能否长期共存，而不互相覆盖、互相泄漏、互相污染。

## 总体架构

Athena 的整体架构可以理解为一条从“外部事件”到“智能体行为”的分层流水线，但它不是单纯的数据管道，而是一个有状态、有边界、有扩展点的长期运行系统。

最上层是 **Koishi 接入层**。它负责把具体平台的消息、事件、通知和适配器输入，转换成 Athena 可以消费的宿主事件。它负责接入和归一化，不负责决策和解释。

中间层是 **Agent Runtime / Agent Loop**。这是 Athena 的执行核心，负责把一次输入转化成一段持续的行动过程，包括是否响应、如何组织上下文、是否调用工具、如何写回结果、何时结束本轮。

运行时紧贴着 **Agent Session** 与 **Session Manager**。前者承担框架层的会话执行职责，后者承担持久化与恢复边界。它们共同保证运行状态不是每轮都从零开始重建。

另一条横向能力线是 **Extension / Plugin SDK**。它提供扩展接入面，用于承接工具、上下文注入、行为修饰、外部集成和附加能力。

**Model Provider** 则是一条独立的资源线。它提供模型适配、能力接入与底层推理资源，但不应吞并 agent、plugin 或业务层的职责。

最贴近 Athena 产品特质的两层，则是 **Context / Memory** 与 **群聊行为决策模块**。前者解决“智能体记得什么、现在看见什么”，后者解决“智能体应不应该说、怎么说、什么时候说”。

如果用一句话概括，Athena 不是“一个大 loop 加很多插件”，而是：

- 接入层负责收事件
- Runtime 负责做行为
- Session 负责保持续
- Manager 负责保可靠
- Provider 负责供模型
- Plugin SDK 负责开边界
- Memory 负责保长期认知
- 行为决策负责保群聊存在感

## 核心模块职责

### Koishi 接入层

Koishi 的职责是宿主适配与事件入口。它负责把平台消息、通知、群事件、私聊事件等输入转成系统内部可处理的事件，但不负责决定是否响应，也不负责拼装业务上下文。

### `packages/agent` / Agent Runtime / Agent Loop

`packages/agent` 应保持为框架层，尽量沿用从 `pi-agent` 迁移来的通用职责，而不向 Athena 的具体业务语义倾斜。

它应该负责的是：

- 通用的 agent loop 运行
- 通用的 session / manager 接口
- 通用的扩展 handler 执行
- 通用的 custom message 流转
- 将扩展提供的上下文与提示词修改能力交给运行时编排

它不应该直接知道：

- 这是群聊还是私聊
- 该不该发言
- 该采用什么群聊策略
- 该如何理解某个频道中的社会关系

### `AgentSession`

`AgentSession` 应尽量维持原状，继续作为框架内的会话执行对象，而不是承担 Athena 的业务会话模型职责。

它更适合负责：

- 从 manager 拉取持久化上下文
- 组织当前轮可供运行时使用的消息与状态
- 收集并运行扩展 handler
- 将 custom message 的渲染交给对应扩展
- 允许扩展直接参与提交给 LLM 的上下文构建与系统提示词修改

它更像一个“会话执行壳”，而不是群聊语义本身。

### `SessionManager`

`SessionManager` 是持久化与恢复边界。它负责状态可靠落盘、历史读取、恢复入口与基本一致性，不负责业务语义解释，也不负责当前轮行为判断。

### Extension / Plugin SDK

扩展层是 Athena 真正的能力开口，但它不是挂在 `agent-session` 外围的附属系统，而应作为 `agent-session` 的内置能力接口进入运行时。

在方向上，它可以明显参考和迁移 `pi-coding-agent` 中已经相对成熟、同时又不过度臃肿的方案。

扩展层至少应支持：

- 工具注册
- 上下文注入与修改
- 系统提示词塑形
- custom message 渲染
- 运行时事件钩子
- 行为层面的局部修饰

但它仍应通过清晰边界进入系统，而不是成为新的混沌入口。

### Model Provider

Model Provider 只负责模型接入、能力适配与底层推理调用，不参与群聊业务判断，也不承担 session 管理职责。它是资源层，不是行为层。

### Context / Memory

这部分应位于 **core 应用层**，而不是 `packages/agent`。它负责 Athena 的群聊上下文、长期记忆、关系记忆、环境状态和历史摘要等业务语义。

### 群聊行为决策模块

这部分同样属于 **core 应用层**。它决定：

- 何时介入
- 是否沉默
- 是否跟进
- 是否插话
- 是否只做观察
- 是否需要结合群聊上下文改变行为策略

也就是说，`packages/agent` 负责“怎么跑”，`core` 负责“跑什么语义”。

## 群聊自主 Agent 行为模型

Athena 的群聊行为模型，不应从“如何生成一句回复”开始设计，而应从“一个长期存在的群成员如何决定自己的行为”开始设计。

一次完整的群聊行为，更像这样一条链：

- 感知发生了什么
- 判断这件事是否与自己有关
- 评估是否值得介入
- 决定介入方式
- 决定是否需要工具或外部能力
- 形成表达
- 观察结果并决定是否跟进

这里最关键的一点是：发言只是行为结果之一，不是唯一目标。

Athena 应把以下行为都视为一等公民：

- 沉默
- 观察
- 延迟回应
- 直接回应
- 轻量接话
- 深度展开
- 跟进补充
- 插话打断
- 转移话题
- 借助工具执行行动
- 只更新内部状态而不对外发言

在行为决策方式上，Athena 不应预设某一条路线永远正确，而应支持多层次决策：

### 第一层：轻量快速判断

适合高频、低成本、低延迟场景，例如：

- 是否被直接提及
- 是否处于私聊
- 是否命中显式触发词
- 当前是否在冷却期
- 当前频道活跃度如何
- Bot 最近是否已经连续发言过多

这一层以规则、阈值、意愿值和启发式为主。目标不是最聪明，而是最快、最便宜、最稳定。

### 第二层：条件触发的 LLM 判断

只在第一层不足以做出可靠决策时，再升级到 LLM 判断，例如：

- 语义上是否真的在询问 Bot
- 当前是否适合插入一句轻量评论
- 某段争论是否需要 Bot 出面调和
- 这次回应更适合玩笑、解释还是保持沉默

这一层应被视为升级路径，而不是默认常开路径。

### 第三层：更重的观察者 / 规划者机制

这类方案更接近 MaiBot、CyberGroupmate 中的高成本路线，例如 planner / responder、Main Agent / Sub Agent、长期观察者等。

它适合用在复杂多轮群聊节奏把握、高价值社会行为判断或更强的长期观察场景中，但不应成为 Athena 当前默认依赖的中心能力。

因此，一个更符合 Athena 的行为方向是：

- 底层默认使用轻量规则与意愿值做快速初判
- 在必要处引入条件触发的 LLM 判断做精修
- 将更重的多阶段观察者模型视作未来增强能力
- 把沉默、延迟、旁观、轻量接话都当作正当行为
- 把“自然存在”而不是“频繁回答”作为最终优化目标

## 记忆与上下文策略

Athena 在记忆与上下文上的设计，不应一开始就追求一个包打天下的复杂核心系统。更合适的路线是：**core 保持最小可用实现，把复杂上下文管理留给插件层。**

这意味着 core 应保留一个最小且可靠的上下文模型：

- 线性的历史拼接
- 基础的系统提示词组装
- 简单的上下文裁剪
- 简单的压缩总结
- 必要的持久化历史读取

也就是说，core 先解决最基础的问题：

- 模型能看到最近发生了什么
- 历史不会无限膨胀
- 会话在长时间运行后仍能继续
- 最基本的上下文连续性能成立

更复杂的能力，例如：

- 长期记忆召回
- 用户画像与关系记忆
- 群聊氛围记忆
- 多层摘要
- 语义检索
- 主题轨迹维护
- 结构化环境状态
- 外部知识缓存
- 个性化上下文拼装

这些都更适合交给插件实现，而不是直接做进 core。

这背后的判断很简单：不同应用场景对“重要上下文”的理解差异极大，没有必要让 core 预设唯一正确答案。Athena 的核心只需要保证“上下文存在”，而不是强行保证“上下文已经足够聪明”。

### 简单压缩内建，复杂压缩外置

上下文压缩是一个例外中的例外，因为只要有长会话，压缩几乎是必需能力。因此 core 应保留简单压缩总结机制。但也只保留到“简单总结”这一层。

更复杂的压缩策略，例如分层摘要、主题导向压缩、面向关系与任务状态的压缩、压缩后结构化检索等，应交给插件层实现。

### 未来的上下文引擎

长期来看，Athena 不一定要把高级上下文能力分散在大量插件里。更进一步，它可以逐渐沉淀出一个独立的“上下文引擎”，专门辅助插件作者设计记忆与上下文策略。

这个引擎的定位不是替代 core，而是服务于插件生态：

- 提供上下文拼装能力
- 提供记忆选择与压缩能力
- 提供可复用的上下文策略组件
- 提供一套比“直接改 prompt”更清晰的抽象

这样可以让 core 保持干净，让插件仍然灵活，同时避免高级上下文能力过早永久焊死进主运行时。

## 与参考项目的关系

Athena 的设计不是从真空中长出来的。它的很多理念、冲动、失败和修正，都能在几个关键参考项目里找到来源。但这些项目对 Athena 的意义，不是“照着实现”，而是帮助它更清楚地知道自己要成为什么，以及不要成为什么。

### YesImBot 早期版本

YesImBot 的 v1、v2、v3 及其延伸尝试，是 Athena 最直接的血缘来源。它们证明了：

- 群聊中的拟人化参与是有价值的
- 意愿值、启发式响应、低成本行为判断有现实意义
- prompt 对角色感和存在感的塑造极其关键
- 轻量但自然的群聊行为，往往比重型但迟缓的系统更符合体验

同时它们也不断提醒：手写 loop、职责混叠、上下文与业务纠缠会迅速毁掉可维护性。

### Letta / MemGPT

Letta 给 Athena 的最大影响，不是某个具体实现细节，而是几种关键观念：

- 记忆不应只等于最近对话
- 系统提示词可以被视为运行时结构
- 思考与表达可以分离
- 长期智能体必须正视上下文窗口限制

Athena 从 Letta 借走的是“长期智能体意识”，而不是它的完整产品形态。

### CyberGroupmate

CyberGroupmate 的参考价值非常高，尤其体现在以下方面：

- 群聊不是问答，而是社会场景
- 发言决策本身是一等能力
- Main Agent / Sub Agent 这类分层视角是成立的
- 记忆、反思和行为组织可以做得比普通聊天 Bot 更深入

它像是 Athena 的一个远景参照物，提醒 Athena 群聊 Agent 可以比“会聊天”走得更远。

### MaiBot

MaiBot 在群聊响应判断上代表了更依赖 LLM 的路线。它的重要意义在于证明：

- “先判断，再发言”是值得投入的
- “不说话”本身可以是显式决策结果
- 响应生成和响应判断未必必须由同一层完成

它与 YesImBot 的纯规则 / 纯意愿值路线形成了鲜明对照，也让 Athena 更清楚地看到：群聊行为决策并不只有一条路。

### `pi-mono` / `pi-coding-agent`

如果说 YesImBot、Letta、CyberGroupmate、MaiBot 更多影响的是理念与产品方向，那么 `pi-mono` / `pi-coding-agent` 影响的就是 Athena 当前最关键的技术收敛方向。

它们提供了：

- 更通用的 agent loop 思路
- 更干净的框架层与业务层分离
- 更成熟但不过度臃肿的扩展能力设计
- 更适合作为 `packages/agent` 技术参照的结构基础

### `promptHosting`

`promptHosting` 不是当前 Athena 的技术模板，但它对项目气质的影响很深。它代表的是 YesImBot / Athena 早期最核心的灵魂层来源：角色存在感、表达方式、人格连续性。

因此，Athena 不能把 prompt 视为可随意替换的边角料。它不是骨架，但它确实是灵魂层的重要组成部分。

### Athena 的差异化方向

把这些参考放在一起看，Athena 实际上在走一条自己的折中路线：

- 继承 YesImBot 的群聊原生感与低成本意识
- 吸收 Letta 的长期智能体观和记忆意识
- 借鉴 CyberGroupmate 的群聊社会性视角
- 参考 MaiBot 的高质量行为判断路线
- 采用 `pi-agent` / `pi-coding-agent` 更成熟的框架思路
- 保留 `promptHosting` 所代表的灵魂层传统

但它最终想形成的，不是这些项目的拼盘，而是一种更稳定的自我定义：

> 一个以群聊为现实起点、以长期数字主体为远期目标、在低成本约束下追求自然社会性参与的智能体框架。

## 分阶段路线图

Athena 的路线图不适合写成“先把所有能力列全，再一项项实现”的功能清单。更合理的方式，是按系统成熟度与边界收敛顺序推进。

### 第一阶段：建立可持续的运行骨架

重点包括：

- 稳定 `packages/agent` 作为通用框架层的边界
- 保持 `agent-session`、`session-manager`、agent loop 的职责清晰
- 让 core 作为 Athena 业务层站稳，而不是反向污染框架层
- 稳定 provider、plugin、agent 三方独立关系
- 让最小上下文、最小压缩、最小工具扩展链先可持续工作

### 第二阶段：收敛群聊行为决策模型

在骨架稳定之后，集中解决 Athena 最核心的产品问题：它如何在群聊里自然地存在。

重点不在“回复得多聪明”，而在：

- 何时回应
- 何时沉默
- 如何控制发言密度
- 如何处理延迟、跟进、插话
- 如何在低成本前提下逐步提升行为判断质量

### 第三阶段：建立插件化的上下文与记忆生态

在行为模型初步成型后，再逐步推进更复杂的上下文与记忆能力。

但仍坚持边界：

- core 保持最小实现
- 高级上下文策略优先插件化
- 记忆、检索、关系建模、环境状态等能力逐步沉淀到插件生态
- 如有必要，再逐步抽象出独立的上下文引擎

### 第四阶段：扩展能力与插件生态成熟

当骨架、行为、上下文都初步稳定后，Athena 才适合真正进入扩展生态建设阶段。

重点会包括：

- 让 Plugin SDK 足够稳定
- 让工具、上下文注入、提示词塑形、行为修饰等能力有清晰边界
- 让 skill、workspace、search、MCP、外部集成等能力在核心外健康生长
- 逐步形成一套对插件作者友好的扩展心智模型

### 第五阶段：从群聊 Agent 走向更一般的数字主体

这是更长期的远景阶段。到这里，Athena 才会逐渐从“服务于群聊环境的大模型插件”继续抽象，走向更一般意义上的数字主体。

这可能包括：

- 逐步淡化对单一聊天平台输入模型的依赖
- 吸收更一般的事件 / 感知输入
- 将行为目标从“聊天回复”扩展到“多种形式的环境参与”
- 借助 Koishi 适配更多平台与媒介
- 最终探索脱离 Koishi 后的独立运行形态

贯穿整个路线图的一条总原则是：

> 先稳骨架，再塑行为；先控复杂度，再长能力；先服务群聊现实，再逐步逼近更一般的数字主体。

## 风险与待决策问题

Athena 当前真正要面对的问题，已经不是“能不能做出来”，而是“能不能在继续生长的同时不再次失控”。

### 主要风险

1. **复杂度再次失控**

   很多能力单看都合理，但只要在边界未稳的情况下同时增长，系统就会再次回到过去那种“所有东西都在变强，但没有哪个部分还能被单独理解和稳定维护”的状态。

2. **框架层被业务层反向污染**

   `packages/agent` 的价值，在于它是一个尽量中立、尽量通用的框架层。如果 core 持续把群聊业务语义压进 agent 框架层，Athena 的长期路线就会被提前锁死。

3. **插件系统再次变成新的混沌入口**

   把复杂能力交给插件层是对的，但插件化不是天然安全。多个插件同时改系统提示词、注入上下文、影响行为或改变工具可见性时，如果没有清晰边界，系统很快会重新失去可预测性。

4. **群聊行为质量与成本约束之间的长期拉扯**

   更自然的行为通常意味着更高的成本、更高的延迟和更高的系统复杂度。这个张力不会一次解决，它会长期存在。

5. **记忆系统过早膨胀**

   长期记忆、关系建模、检索增强、群氛围状态、多层摘要等能力都很诱人，但也最容易把 core 重新拖进复杂度黑洞。

6. **对远景的过度提前实现**

   “赛博灵魂”是长期目标，但如果为了未来的独立主体形态而在今天就提前引入过重抽象，反而会稀释当前最清晰、最有验证价值的群聊主战场。

### 需要持续判断的问题

以下问题现在不需要全部定死，但应当被明确记录：

1. 群聊响应判断最终会停留在多大程度的启发式与规则驱动？
2. LLM 参与行为判断应当是默认常开，还是条件升级路径？
3. 插件可以在多大程度上修改系统提示词与 LLM 可见上下文？
4. 高级上下文能力会长期保持插件化，还是逐渐沉淀为独立上下文引擎？
5. provider 原生能力与普通扩展能力的边界未来是否需要更清晰抽象？
6. Athena 何时才真正值得从群聊主战场扩展到更一般的感知与行动场景？
7. 未来若真的脱离 Koishi，哪些当前边界设计已经足够支撑，哪些只是暂时性适配？

## 总结

Athena 真正要面对的挑战，不是某个单独功能怎么实现，而是在持续增长行为能力、插件能力、上下文能力与模型能力的同时，始终守住系统边界，不再重演“能力越来越多，但整体越来越不可维护”的历史。

它当前最重要的任务，不是尽快把所有高级能力都做出来，而是先建立一个真正值得继续生长的骨架：

- 让框架层保持框架层
- 让业务层承担业务语义
- 让扩展层负责复杂能力增长
- 让群聊行为成为真正的一等问题
- 让远景指导方向，而不是提前吞噬实现

如果这条路线能够成立，那么 Athena 才有机会从一个服务于群聊环境的大模型插件，逐步成长为一个更一般意义上的数字主体。
