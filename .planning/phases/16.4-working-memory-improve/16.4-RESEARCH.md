# Phase 16.4: working-memory-improve - Research

**Researched:** 2026-02-22
**Domain:** Agent loop working memory management (context trimming, timeline recording)
**Confidence:** HIGH

## Summary

This phase transforms the ThinkActLoop's messages array from unbounded linear growth into a budget-controlled working memory with progressive degradation. The current implementation (`loop.ts`) appends assistant responses and tool results to a `messages` array without any trimming — each round adds two messages (assistant + user with tool results), and the only size control is `maxToolResultLength` which truncates individual tool results at creation time.

The phase has two main axes: (1) a trimming system that applies two-level degradation (softTrim → hardClear) to older rounds' tool results and assistant replies before each LLM call, and (2) replacing the post-loop `AgentSummary` timeline event with per-round `AgentResponse` events that capture complete data before trimming occurs.

**Primary recommendation:** Implement trimming as a standalone function that mutates the messages array in-place before each `modelService.call()`, operating on character budget with configurable keep-last-N-rounds protection. Replace `AgentSummary` with `AgentResponse` across horizon types, EventManager, and observation pipeline.

### Locked Decisions

**Trimming strategy:**
- Check messages total before each round; trim if over threshold
- Two-level progressive degradation: softTrim (keep head/tail chars, ellipsis middle) → hardClear (replace with placeholder, tool name only)
- Keep last N rounds complete (configurable, default 2)

- Write per-round in real-time (one event per round), don't wait for loop end
- Trimming only affects in-memory messages array, not timeline records

- Character count estimation, no tokenizer dependency
- Both tool results and assistant replies can be trimmed

### Pattern 1: Progressive Trimming Function
**What:** A pure function `trimMessages(messages, config)` that scans the messages array and applies degradation based on round age relative to the end.
**When to use:** Called at the top of each while-loop iteration, before `modelService.call()`.
**Example:**
```typescript
// core/src/services/agent/trimmer.ts

interface TrimConfig {
  charBudget: number;       // e.g. 30000
  keepLastRounds: number;   // e.g. 2
  softTrim: { headChars: number; tailChars: number };  // e.g. 800, 800
}

interface LoopMessage {
  role: "user" | "assistant";
  content: string;
  /** Set by trimmer to track degradation state */
  _trimState?: "none" | "soft" | "hard";
}

function trimMessages(messages: LoopMessage[], config: TrimConfig): void {
  // 1. Calculate total chars
  // 2. If under budget, return
  // 3. Identify round boundaries (assistant+user pairs)
  // 4. Skip first message (initial user context) and last N rounds
  // 5. For eligible rounds, apply softTrim to tool-result user messages first
  // 6. If still over budget, apply hardClear to tool-result user messages
  // 7. If still over budget, softTrim/hardClear assistant messages
}
```

### Pattern 2: Per-Round Timeline Recording
**What:** After each round's tool execution completes, immediately write an `AgentResponse` event to the timeline before any trimming occurs.
**When to use:** Inside the while-loop, after `executeActions()` returns and before the next iteration's trim.
**Example:**
```typescript
// In loop.ts, after executeActions():
await horizon.events.recordAgentResponse({
  scope: percept.scope,
  timestamp: new Date(),
  data: {
    round,
    assistantText: rawText,
    actions: response.actions,
    toolResults,
  },
});
// Then on next iteration, trimMessages() may degrade this round's content in memory
```

### Pattern 3: Layered Context in HorizonView
**What:** The horizon-view template gains a `<working-memory>` section. The `formatHorizonText()` method receives working-memory data alongside history. Agent actions in history show only summary lines; full details go to working-memory.
**When to use:** When rendering the user message content for LLM calls.

### Anti-Patterns to Avoid
- **Trimming inside formatToolResults():** Don't mix formatting with budget management. Trimming is a separate concern that operates on the full messages array.
- **Storing trimmed data in timeline:** Timeline must always receive the complete, untrimmed data. Trimming is ephemeral and only affects the in-memory messages array.
- **Token-based budget:** User explicitly chose character-based counting. Don't introduce tokenizer dependencies.
- **Trimming the initial user message (index 0):** The first message contains the horizon view context (environment, members, history). It must never be trimmed by the working-memory trimmer.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Round boundary detection | Complex index math | Convention: rounds are (assistant, user) pairs after index 0 | Messages array has predictable structure: [user₀, assistant₁, user₁, assistant₂, user₂, ...] |
| Timeline ID generation | uuid/nanoid | `Random.id()` from koishi | Project standard, already used everywhere |

**Key insight:** The messages array structure is fully controlled by the loop — it always alternates assistant/user after the initial user message. This makes round detection trivial (pairs from index 1 onward).

## Common Pitfalls

### Pitfall 1: Trimming Before Recording
**What goes wrong:** If trimMessages() runs before the current round's data is written to timeline, the timeline gets degraded data.
**Why it happens:** Natural instinct to trim at the top of the loop, but recording happens at the bottom.
**How to avoid:** Record to timeline immediately after executeActions(), then trim at the start of the next iteration (or trim only the messages that existed before the current round).
**Warning signs:** Timeline entries with "...(trimmed)" or placeholder text.

### Pitfall 2: Off-by-One in Round Counting
**What goes wrong:** The "keep last N rounds" logic miscounts because the initial user message (index 0) is not part of any round.
**Why it happens:** Messages array starts with a user message that isn't a tool-result message.
**How to avoid:** Define rounds as pairs starting from index 1: round 1 = messages[1]+messages[2], round 2 = messages[3]+messages[4], etc. The initial user message at index 0 is always protected.
**Warning signs:** The most recent round gets trimmed, or one too many old rounds are preserved.

### Pitfall 3: Double Trimming
**What goes wrong:** A message that was already softTrimmed gets softTrimmed again, producing nested ellipsis or corrupted head/tail boundaries.
**Why it happens:** trimMessages() runs every iteration without tracking what's already been trimmed.
**How to avoid:** Track trim state per message (e.g., `_trimState` field). softTrim only applies to "none" state; hardClear applies to "none" or "soft" state.
**Warning signs:** Messages with `...(...(trimmed)...)` patterns.

### Pitfall 4: AgentSummary Removal Breaking Observation Pipeline
**What goes wrong:** Removing `AgentSummary` type breaks `toObservations()`, `formatObservation()`, and the `Observation` union type.
**Why it happens:** These types are used in the history rendering pipeline (horizon-view template).
**How to avoid:** Replace AgentSummary with AgentResponse everywhere in the type chain: `TimelineEventType`, `TimelineEntry`, `Observation`, `toObservations()`, `formatObservation()`. The history summary line format changes from `summary` string to computed summary from AgentResponse data.
**Warning signs:** TypeScript errors in horizon service, or history showing raw JSON instead of summary lines.

### Pitfall 5: Working-Memory Section in Wrong Message
**What goes wrong:** Working-memory details end up in the system prompt instead of the user message.
**Why it happens:** Confusion between system prompt (static: identity/style/memories) and user message (dynamic: horizon-view).
**How to avoid:** Working-memory is part of the horizon-view template (user message), not a prompt injection point. The data flows through `formatHorizonText()`.
**Warning signs:** System prompt growing with each round, or working-memory appearing in cached prompt sections.

## Code Examples

### softTrim Implementation
```typescript
function softTrim(text: string, headChars: number, tailChars: number): string {
  if (text.length <= headChars + tailChars + 20) return text;
  return text.slice(0, headChars) + "\n...(trimmed)...\n" + text.slice(-tailChars);
}
```

### hardClear for Tool Results
```typescript
function hardClearToolResult(userMessage: string): string {
  // Tool result messages follow format: "Tool results:\n[{...}]"
  // Extract tool names from the JSON, replace with placeholder
  try {
    const jsonStart = userMessage.indexOf("[");
    if (jsonStart < 0) return "[tool results cleared]";
    const results = JSON.parse(userMessage.slice(jsonStart));
    const names = results.map((r: { name: string }) => r.name).join(", ");
    return `[tool results cleared: ${names}]`;
  } catch {
    return "[tool results cleared]";
  }
}
```

// In formatObservation(), for AgentResponse observations:
function formatAgentActionSummary(data: AgentResponseData): string {
  const toolNames = data.actions.map(a => a.name);
  const sendAction = data.actions.find(a => a.name === "send_message");
  if (sendAction) {
    const content = (sendAction.params as { content?: string })?.content ?? "";
    // Show sent content in history (agent speech equivalent)
    const otherTools = toolNames.filter(n => n !== "send_message");
    const toolPart = otherTools.length ? ` [also: ${otherTools.join(", ")}]` : "";
    return `[Bot]: ${content}${toolPart}`;
  }
  return `[Bot Action]: ${toolNames.join(", ")}`;
}

## Open Questions

1. **softTrim default values**
   - What we know: OpenClaw uses headChars=1500, tailChars=1500 with maxChars=4000
   - Recommendation: Use headChars=800, tailChars=800 as defaults. Our tool results are already pre-truncated to `maxToolResultLength` (default 4000), so the softTrim operates on already-bounded content. Smaller head/tail is sufficient.

2. **Character budget default value**
   - What we know: The initial user message (horizon-view) can be 2000-5000 chars depending on history length. Each round adds ~5000-10000 chars (assistant JSON + tool results).
   - Recommendation: Default budget of 30000 characters. This allows ~3-4 full rounds plus the initial context, with trimming kicking in for longer loops.

3. **Working-memory data flow through formatHorizonText**
   - What we know: Currently `formatHorizonText(view: HorizonView)` only receives the HorizonView. Working-memory data comes from the loop's messages array, not from horizon.
   - Recommendation: The loop should build working-memory formatted strings and pass them to `formatHorizonText()` via an extended parameter or a new field on HorizonView. Alternatively, the working-memory section could be appended directly in the loop after `formatHorizonText()` returns, keeping horizon unaware of working-memory. The latter is simpler and maintains separation of concerns — horizon handles history, the loop handles its own working memory.

4. **Wrap-up round (maxRounds exceeded) and trimming**
   - What we know: The current loop has a special wrap-up path when `round >= maxRounds` that appends messages and makes one final call.
   - Recommendation: The wrap-up path should also apply trimming before its final `modelService.call()`. The trimmer function is stateless and can be called at any point.
