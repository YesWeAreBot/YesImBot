## Summary

This phase restructures the boundary between `horizon` and `agent` modules. Currently, `horizon/listener.ts` does three things: (1) records timeline events, (2) builds Percept objects, and (3) runs an aggregation window with priority-based trigger selection. After this phase, horizon only does (1) record + broadcast raw events. The agent module takes over (2) Percept construction and (3) aggregation windowing.

Simultaneously, the prompt rendering pipeline changes. Currently, `loop.ts` calls `horizon.buildView()` + `horizon.toStructured()` + manual `envLines` string concatenation to build the system prompt's environment section. After this phase, the system prompt becomes purely static (identity/style/memories), and all dynamic context (environment, members, history, trigger) moves into a single user message rendered via `formatHorizonText()` + `horizon-view.mustache`.

**Primary recommendation:** Execute as a big-bang refactor touching 6-8 files. The changes are tightly coupled (event format change + consumer change + template change), so incremental migration would create more complexity than it saves.

## Architecture Patterns

### Current Data Flow (BEFORE)

```
koishi middleware
  → listener.recordUserMessage()     [timeline write]
  → listener.buildPercept()          [Percept construction]
  → listener.schedulePercept()       [aggregation window + priority override]
  → ctx.emit("horizon/percept")      [broadcast complete Percept]
  → agent.handlePercept()            [willingness gate]
  → loop.run()                       [buildView + toStructured + envLines → system prompt]
```

### Target Data Flow (AFTER)

```
koishi middleware
  → listener.recordUserMessage()     [timeline write — unchanged]
  → ctx.emit("horizon/event", ...)   [broadcast raw event payload]
  → agent.handleEvent()              [willingness per-event + aggregation window]
  → agent constructs Percept          [from last event in window]
  → loop.run()                       [buildView + formatHorizonText → user message]
```

### Pattern 1: Horizon Event Type System

The codebase already uses koishi's declaration merging for `horizon/percept`. The new pattern replaces this with `horizon/event`:

```typescript
// horizon/types.ts — event map + declaration merging
interface HorizonEventMap {
  "horizon/message": HorizonMessageEvent;
  // future: "horizon/reaction": HorizonReactionEvent;
}

declare module "koishi" {
  interface Events extends HorizonEventMap {}
}

interface HorizonMessageEvent {
  scope: Scope;
  timestamp: Date;
  payload: {
    messageId: string;
    senderId: string;
    senderName: string;
    content: string;
  };
  triggerType: TriggerType;
  runtime?: { session: Session };
}
```

This follows the existing koishi pattern at `listener.ts:9-13` and the locked decision to use event mapping + declaration merging.

### Pattern 2: Aggregation Window in Agent

Current `schedulePercept()` in listener uses priority-based override (higher trigger priority wins). The new design uses "last event wins" semantics:

```typescript
// agent/service.ts — aggregation window
interface PendingWindow {
  cancelTimer: () => void;
  events: HorizonMessageEvent[];  // accumulate all events in window
}

// On each horizon/event:
// 1. Compute willingness independently
// 2. If shouldReply, add to window
// 3. Reset timer to configurable window duration
// 4. On timer expiry: construct Percept from last event, run loop
```

### Pattern 3: User Message Template Rendering

Current `loop.ts:39-65` manually builds envLines. The new approach delegates entirely to `formatHorizonText()`:

```typescript
// loop.ts — simplified
const view = await horizon.buildView(percept);
const userContent = horizon.formatHorizonText(view);
const messages = [{ role: "user", content: userContent }];
```

The `horizon-view.mustache` template already has the right structure but needs enhancement for the trigger/history split based on `New`/`Active` stage.

### Pattern 4: Static System Prompt

The system prompt removes the `{{>environment}}` partial. Only static sections remain:

```mustache
{{>identity}}

{{>style}}

{{>core_memories}}

{{>working_memory}}

{{>extra}}
```

The `renderToString("system", ...)` call in loop.ts no longer needs `view`, `environment_content`, or `has_environment` in its scope.

### Anti-Patterns to Avoid
- **Keeping toStructured() "just in case":** Decision is to delete it entirely. Dead code invites confusion.
- **Passing Session through Percept for non-runtime use:** Session should only be in `runtime` field for send operations, not for data extraction (that's horizon's job during recording).
- **Aggregation window in horizon:** This is a scheduling/decision concern, not a data layer concern.

## Timeline Stage Split (History vs Trigger)

This is the most nuanced implementation detail. Currently:

1. `listener.recordUserMessage()` writes with `stage: TimelineStage.New`
2. `buildView()` queries ALL entries (no stage filter) via `events.query()`
3. After loop completes, `events.markAsActive()` promotes `New` → `Active`

The locked decision says: history = Active messages, trigger = New messages.

**Current gap:** `buildView()` doesn't filter by stage, and `Observation` type doesn't carry stage info. The `events.query()` in `buildView()` returns all entries regardless of stage.

**Implementation approach:**
1. Add `stage` field to `Observation` types (or at least `MessageObservation`)
2. In `toObservations()`, propagate `entry.stage` to the observation
3. In `formatHorizonText()`, split observations: `stage === 'active'` → history section, `stage === 'new'` → trigger section
4. The trigger section replaces the current hardcoded `triggerType`/`triggerMessage` template variables

This means `formatHorizonText()` no longer needs `triggerType`/`triggerMessage` from the Percept — it gets trigger info directly from New-stage timeline entries. The Percept's trigger info becomes metadata for the agent's decision-making, not for prompt rendering.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Template rendering | Custom string concatenation | `formatHorizonText()` + Mustache | Already exists, tested, single source of truth |
| Event typing | Manual type guards | Declaration merging + event map | Koishi's own pattern, type-safe |
| Timer management | Raw setTimeout | `ctx.setTimeout` | Koishi lifecycle-aware, auto-cleanup on dispose |

**Key insight:** The entire `envLines` manual construction in `loop.ts` (20 lines) is being replaced by a single `formatHorizonText()` call. This is the core "don't hand-roll" win of this phase.

## Common Pitfalls

### Pitfall 1: Stale Percept in Deferred Judgment
**What goes wrong:** Deferred judgment fires with a Percept constructed minutes ago. The `buildView()` call fetches current timeline, but the Percept's payload references an old message.
**Why it happens:** Aggregation window constructs Percept at window-close time, but deferred judgment runs much later.
**How to avoid:** Deferred judgment should re-query timeline at execution time (it already does via `buildView()`). The Percept's payload is only used for scope identification, not content rendering.
**Warning signs:** Deferred judgment responding to messages that are no longer contextually relevant.

### Pitfall 2: Race Between Record and Broadcast
**What goes wrong:** Agent receives `horizon/event`, calls `buildView()`, but the message hasn't been written to timeline yet.
**Why it happens:** If `recordUserMessage()` is async and `ctx.emit()` fires before the DB write completes.
**How to avoid:** Locked decision explicitly states: "horizon 先记录到 timeline，再广播事件". The middleware must `await recordUserMessage()` before emitting. Current code already does this correctly.
**Warning signs:** `buildView()` returning history that doesn't include the triggering message.

### Pitfall 3: Import Cycles After Type Migration
**What goes wrong:** Moving Percept types to agent creates circular imports if horizon still needs them.
**Why it happens:** `HorizonView` currently references `Percept`. If Percept moves to agent, horizon imports from agent, and agent imports from horizon.
**How to avoid:** `HorizonView.percept` field type needs adjustment. Options:
- (a) Make `HorizonView` generic or use a minimal interface that both modules satisfy
- (b) Keep a slim `BasePercept` in horizon/types, extend it in agent
- (c) Move `HorizonView` to also live in agent (but horizon.buildView returns it)
- Recommended: (b) — keep `Scope` + minimal percept identity in horizon, full `UserMessagePercept` in agent

### Pitfall 4: Forgetting to Update PromptService Registration
**What goes wrong:** `environment` partial still registered, `environment` injection point still active, ghost content appears in system prompt.
**Why it happens:** PromptService constructor registers all partials including environment.
**How to avoid:** Remove environment from `partialMap` in constructor AND remove `{{>environment}}` from `system.mustache`. Both must happen together.

### Pitfall 5: Aggregation Window Semantics Change
**What goes wrong:** Current "highest priority wins" is replaced by "last event wins", but willingness was computed on earlier events that may have been the actual trigger.
**Why it happens:** Semantic mismatch between "which event triggered willingness" and "which event is the main trigger".
**How to avoid:** Per the locked decision, each event independently computes willingness. The window only controls timing. The last event is the main trigger because it's what the bot is "responding to" in conversation flow. This is intentional — "语义更符合对话直觉（回应最新的消息）".

### Agent Aggregation Window (service.ts)

```typescript
private handleEvent(event: HorizonMessageEvent): void {
  const channelKey = `${event.scope.platform}:${event.scope.channelId}`;

  // Willingness computed per-event
  const result = this.willingness.processMessage(
    channelKey, event.triggerType, event.payload.content,
  );

  if (!result.shouldReply) {
    // deferred judgment logic (unchanged concept)
    return;
  }

  // Aggregation: direct messages skip window
  if (event.scope.isDirect) {
    const percept = this.buildPercept(event);
    this.enqueue(channelKey, percept);
    return;
  }

  // Group: reset window timer, keep latest event
  const existing = this.pendingWindows.get(channelKey);
  if (existing) existing.cancelTimer();

  const cancel = this.ctx.setTimeout(() => {
    this.pendingWindows.delete(channelKey);
    const percept = this.buildPercept(event); // last event wins
    this.enqueue(channelKey, percept);
  }, this.config.aggregationWindow ?? 1500);

  this.pendingWindows.set(channelKey, { cancelTimer: cancel, lastEvent: event });
}
```

## Open Questions

1. **HorizonView.percept field after type migration**
   - What we know: `HorizonView` is defined in horizon/types.ts and references `Percept`. Percept moves to agent.
   - What's unclear: Exact strategy to avoid circular imports.
   - Recommendation: Keep a minimal `BasePercept` (id + type + scope + timestamp) in horizon/types. Agent extends it. `HorizonView` references `BasePercept`. `buildView()` accepts `UserMessagePercept` (which extends `BasePercept`).

2. **Observation stage propagation**
   - What we know: `toObservations()` currently doesn't propagate `stage` from timeline entries.
   - What's unclear: Whether to add `stage` to `Observation` type or handle splitting at the query level.
   - Recommendation: Add optional `stage` to `MessageObservation`. Propagate in `toObservations()`. Split in `formatHorizonText()`.

3. **Environment injection point lifecycle**
   - What we know: Environment partial is removed from system.mustache.
   - What's unclear: Whether to remove `"environment"` from `INJECTION_POINTS` entirely.
   - Recommendation: Keep the injection point in the type system but remove the default partial registration. Future phases (Skill system) may want to inject environment-related content.
