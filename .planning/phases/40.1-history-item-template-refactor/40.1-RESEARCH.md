# Phase 40.1: History Item 模板化与渲染格式精简 - Research

**Researched:** 2026-02-28
**Domain:** Mustache partial templating, formatObservation refactor, XML rendering format — internal TypeScript refactoring
**Confidence:** HIGH (all findings from direct codebase inspection)

## Summary

Phase 40.1 is a pure rendering refactoring phase. Phase 40 established a unified XML render pipeline where `formatObservation()` returns raw XML strings that are pushed into a `string[]` array and iterated in `horizon-view.mustache` via `{{{.}}}`. Phase 40.1 moves the rendering logic into a Mustache partial (`history-item.mustache`) so that `formatObservation()` returns a structured data object instead of a pre-rendered string, and `horizon-view.mustache` delegates to `{{>history-item}}` for each entry.

Simultaneously, the XML format is simplified: `<bot-action>` loses its `round` and `trigger` attributes (causal order is now expressed by timeline position), `<bot-action>` is renamed to `<action>`, and `<msg>` sender information moves from XML attributes into inline text content. Time format changes from `HH:MM` to `DD:HH:MM` (day-hours-minutes relative format).

The scope is tightly bounded: two template files, one method in `service.ts`, and the `formatHorizonText` scope construction. No new npm packages, no DB changes, no changes to `types.ts` or `manager.ts`.

**Primary recommendation:** Single plan — update `formatObservation()` to return a data object, update `formatHorizonText` to pass a partials map to `Mustache.render`, write `history-item.mustache` with all observation type branches, update `horizon-view.mustache` to use `{{>history-item}}`.

## Architecture Patterns

### Current State (Phase 40 output — what exists today)

```
formatObservation(obs, selfId, channelKey): string
  message     → '<msg id="N" sender="X" senderId="Y" time="HH:MM">content</msg>'
  agent.action → '<bot-action round="N" trigger="#M">lines</bot-action>'
  agent.response (error) → '<bot-error round="N">error</bot-error>'
  agent.response (ok)    → ''  (filtered at call site)

formatHorizonText(view, percept?): string
  historyObs: string[] = []
  for obs in view.history:
    formatted = formatObservation(obs, ...)
    if formatted: historyObs.push(formatted)
  scope = { history: historyObs, ... }
  Mustache.render(horizonViewTpl, scope)   ← NO partials map passed

horizon-view.mustache:
  <history>
  {{#history}}
  {{{.}}}          ← raw string interpolation, no partial
  {{/history}}
  </history>

history-item.mustache (current, incomplete):
  {{#is_message}}
  <message>[{{id}}|{{times}}|{{senderName}}({{senderId}})] {{content}}</message>
  {{/is_message}}
  ← only handles message type, uses old format, not wired into horizon-view
```

### Target State (after Phase 40.1)

```
formatObservation(obs, selfId, channelKey): HistoryItemData  ← returns object, not string

HistoryItemData (discriminated union via boolean flags for Mustache):
  is_message: boolean
  is_action: boolean
  is_error: boolean
  // message fields
  id?: number           (shortId)
  time?: string         (DD:HH:MM relative format)
  senderLine?: string   (inline: "SenderName(senderId)")
  replyLine?: string    (inline: "[回复: quoteId quoteContent]")
  content?: string
  // action fields
  actionContent?: string  (tool call summary lines)
  // error fields
  errorContent?: string

formatHorizonText(view, percept?): string
  historyItems: HistoryItemData[] = []
  for obs in view.history:
    item = formatObservation(obs, ...)
    if item: historyItems.push(item)
  scope = { history: historyItems, ... }
  partials = { 'history-item': historyItemTpl }
  Mustache.render(horizonViewTpl, scope, partials)  ← partials map passed

horizon-view.mustache:
  <history>
  {{#history}}
  {{>history-item}}    ← partial reference
  {{/history}}
  </history>

history-item.mustache:
  {{#is_message}}<msg id="{{id}}" time="{{time}}">{{senderLine}}{{#replyLine}} {{replyLine}}{{/replyLine}} {{content}}</msg>{{/is_message}}
  {{#is_action}}<action>{{actionContent}}</action>{{/is_action}}
  {{#is_error}}<error>{{errorContent}}</error>{{/is_error}}
```

### Pattern 1: Mustache Partials with `Mustache.render`

The third argument to `Mustache.render` is a partials map:

```typescript
// Source: mustache.js official API
Mustache.render(template, view, partials);
// partials: Record<string, string>

// In formatHorizonText:
const partials = { "history-item": this.historyItemTpl };
const rendered = Mustache.render(this.horizonViewTpl, scope, partials).trim();
```

The `history-item.mustache` template is loaded once (lazy, same pattern as `horizonViewTpl`) and cached on the service instance.

**Important:** `Mustache.escape` is set to identity (`text => text`) in `renderer.ts` — this means Mustache does NOT HTML-escape values. The existing `esc()` helper in `formatObservation` handles XML attribute escaping manually. For Phase 40.1, content values passed to the template must already be safe (pre-escaped by ElementFormatterService at receive time).

### Pattern 2: Mustache Section Booleans for Type Dispatch

Mustache has no `if/else` — type dispatch uses boolean section flags:

```mustache
{{#is_message}}...{{/is_message}}
{{#is_action}}...{{/is_action}}
{{#is_error}}...{{/is_error}}
```

The data object sets exactly one flag to `true`. Mustache treats truthy values as section-open. This is the standard Mustache pattern for discriminated rendering.

```typescript
// message observation:
return { is_message: true, is_action: false, is_error: false, id, time, senderLine, content, ... }

// action observation:
return { is_message: false, is_action: true, is_error: false, actionContent }

// error observation:
return { is_message: false, is_action: false, is_error: true, errorContent }
```

### Pattern 3: Relative Time Format `DD:HH:MM`

Success criterion 3 specifies `time="DD:HH:MM"` format. This is a relative time (days:hours:minutes since some epoch or since the start of the conversation). The most natural interpretation given the context is elapsed time since the observation's timestamp relative to "now" or relative to the first message in the window.

However, looking at the success criteria more carefully: `<msg id="N" time="DD:HH:MM">` — this is likely a compact absolute timestamp where DD=day-of-month, HH=hour, MM=minute. This avoids the complexity of computing relative offsets and is consistent with the existing `HH:MM` format (just adding the day component for cross-day conversations).

```typescript
// DD:HH:MM — day of month, hour, minute
const d = obs.timestamp;
const dd = String(d.getDate()).padStart(2, "0");
const hh = String(d.getHours()).padStart(2, "0");
const mm = String(d.getMinutes()).padStart(2, "0");
const time = `${dd}:${hh}:${mm}`;
```

This is a LOW confidence interpretation — the success criteria says "相对时间" (relative time) but the format `DD:HH:MM` could mean either. The planner should clarify or pick the simpler absolute interpretation.

### Pattern 4: Inline Sender Format

Success criterion 3: `SenderName(senderId) content`
Success criterion 4: `SenderName(senderId) [回复: quoteId quoteContent] content`

The sender info moves from XML attributes into the text content of `<msg>`. This means:

- `sender` and `senderId` attributes are removed from `<msg>`
- `id` and `time` remain as attributes (structural metadata)
- Content becomes: `senderLine + optional replyLine + content`

```typescript
// senderLine construction (same logic as current senderName/senderId):
const isBot = selfId && obs.sender.id === selfId;
const senderName = isBot ? "[Bot]" : `${badge}${obs.sender.name}`;
const senderId = isBot ? "bot" : obs.sender.id;
const senderLine = `${senderName}(${senderId})`;

// replyLine construction:
let replyLine = "";
if (obs.replyTo && channelKey) {
  const replyShortId = this.getShortId(channelKey, obs.replyTo);
  if (replyShortId !== undefined) {
    // quoteContent not available in MessageObservation — only quoteId
    replyLine = `[回复: ${replyShortId}]`;
  }
}
```

Note: `MessageObservation` has `replyTo?: string` (the native message ID of the quoted message) but does NOT carry the quoted content. The success criterion says `[回复: quoteId quoteContent]` — but `quoteContent` is not in the current data model. This is a gap the planner must address: either accept `[回复: quoteId]` without content, or check if `MessageEventData` / `MessageObservation` needs a `replyContent` field added.

### Pattern 5: `<action>` and `<tool>` Tag Rename

Success criteria 1 and 2:

- `<bot-action>` → `<action>` (remove `round` and `trigger` attributes)
- `<bot-tool>` → `<tool>` (if it exists — currently no `<bot-tool>` tag in the codebase)

The `round` and `trigger` attributes are removed because the unified timeline makes causal order implicit (actions follow the messages that triggered them). The content of the action tag stays the same (tool call summary lines).

```mustache
{{#is_action}}<action>{{actionContent}}</action>{{/is_action}}
```

### Anti-Patterns to Avoid

- **Passing `history` as `string[]` to the template:** After this phase, `history` must be an array of objects (HistoryItemData), not strings. The `{{{.}}}` raw interpolation in `horizon-view.mustache` must be replaced with `{{>history-item}}`.
- **Forgetting to pass partials to `Mustache.render`:** The current `Mustache.render(this.horizonViewTpl, scope)` call has no third argument. Without the partials map, `{{>history-item}}` renders as empty string silently — no error thrown.
- **Caching `horizonViewTpl` but not `historyItemTpl`:** Both templates should be lazily loaded and cached on the service instance with the same `??=` pattern.
- **Mustache auto-escaping:** `Mustache.escape` is set to identity in `renderer.ts` (line 5: `Mustache.escape = (text) => text`). This applies globally. Content values in the template data object are NOT auto-escaped by Mustache. The `esc()` helper is still needed for any values that go into XML attribute positions — but since Phase 40.1 moves sender info into content (not attributes), the escaping concern shifts. Content is pre-escaped by ElementFormatterService; senderName/senderId are not user-controlled content but should still be safe.
- **Breaking the `triggerObs` / `historyObs` split:** `formatHorizonText` currently splits observations into `historyObs` (past) and `triggerObs` (new messages). The `trigger` section in the template uses `{{{.}}}` raw interpolation. After Phase 40.1, `trigger` items are also message observations — they should use the same `{{>history-item}}` partial. The `trigger` array must also become `HistoryItemData[]`.

## Common Pitfalls

### Pitfall 1: `Mustache.render` Silently Ignores Unknown Partials

**What goes wrong:** If `{{>history-item}}` is in the template but no partials map is passed to `Mustache.render`, the partial renders as empty string. No error is thrown. The history section appears empty in the LLM prompt.
**Why it happens:** Mustache's partial resolution is optional — missing partials are silently skipped.
**How to avoid:** Always pass `{ 'history-item': this.historyItemTpl }` as the third argument to `Mustache.render` in `formatHorizonText`. Add a test that verifies history items appear in the rendered output.
**Warning signs:** `<history>` block is empty in rendered prompt despite `view.history` having entries.

### Pitfall 2: `trigger` Section Also Needs HistoryItemData Objects

**What goes wrong:** `formatHorizonText` splits observations into `historyObs` and `triggerObs`. After Phase 40.1, both arrays must contain `HistoryItemData` objects (not strings). The `trigger` section in `horizon-view.mustache` currently uses `{{{.}}}` — it must also switch to `{{>history-item}}`.
**Why it happens:** The refactor touches `history` but it's easy to forget `trigger` uses the same rendering path.
**How to avoid:** Update both `history` and `trigger` sections in `horizon-view.mustache` to use `{{>history-item}}`. Update both `historyItems` and `triggerItems` arrays in `formatHorizonText` to hold `HistoryItemData`.
**Warning signs:** TypeScript compile error when pushing `HistoryItemData` into a `string[]` — catches this at compile time if types are updated.

### Pitfall 3: `quoteContent` Not Available in MessageObservation

**What goes wrong:** Success criterion 4 specifies `[回复: quoteId quoteContent]` but `MessageObservation` only has `replyTo?: string` (the native message ID). The quoted content is not stored in the timeline.
**Why it happens:** Phase 33 added `replyTo` as a message ID reference, not a content copy.
**How to avoid:** Two options: (a) accept `[回复: quoteId]` without content for now, or (b) add `replyContent?: string` to `MessageEventData` and `MessageObservation`. Option (a) is simpler and still satisfies the spirit of the criterion. The planner should decide.
**Warning signs:** If `replyContent` is added to the data model, `listener.ts` must populate it from `session.quote.content` at record time.

### Pitfall 4: `formatObservation` Return Type Change Breaks Callers

**What goes wrong:** `formatObservation` currently returns `string`. Changing it to return `HistoryItemData` (or `HistoryItemData | null`) will break any caller that expects a string.
**Why it happens:** The method is called in `formatHorizonText` and potentially in tests.
**How to avoid:** Check all callers of `formatObservation` before changing the signature. Current callers: only `formatHorizonText` in `service.ts`. The test file `format-horizon-text.test.ts` tests `formatHorizonText` indirectly via `Mustache.render` — it does not call `formatObservation` directly. Safe to change.
**Warning signs:** TypeScript compile error `Type 'HistoryItemData' is not assignable to type 'string'`.

### Pitfall 5: `history-item.mustache` Not Registered as a Retained Partial

**What goes wrong:** `PromptService` constructor registers `horizon-view` as a retained partial (line 56: `this.registerPartial("horizon-view", this.loadPartial("horizon-view"))`). The `history-item` partial is NOT registered there. `HorizonService.formatHorizonText` loads `horizon-view` via `loadPartial` and passes it directly to `Mustache.render`. The `history-item` partial must be loaded separately and passed in the partials map.
**Why it happens:** The `PromptService.partials` map is used by `MustacheRenderer.render` (the injection-point renderer), not by `HorizonService`'s direct `Mustache.render` call. These are two separate rendering paths.
**How to avoid:** In `HorizonService`, add a `private historyItemTpl?: string` field. Load it lazily in `formatHorizonText` with `this.historyItemTpl ??= this.ctx["yesimbot.prompt"].loadPartial("history-item")`. Pass it in the partials map to `Mustache.render`.

  is_message: boolean;
  is_action: boolean;

## Open Questions

1. **`DD:HH:MM` — absolute day or relative elapsed time?**
   - What we know: Success criteria says "时间格式改为相对时间" (relative time format) and shows `DD:HH:MM`
   - What's unclear: Is DD the day-of-month (absolute) or days-elapsed since conversation start (relative)?
   - Recommendation: Use day-of-month (`d.getDate()`) for simplicity. True relative time requires tracking conversation start time, which adds complexity. The planner should confirm.

2. **`quoteContent` in reply line**
   - What we know: Success criterion 4 shows `[回复: quoteId quoteContent]` but `MessageObservation.replyTo` is only the native message ID
   - What's unclear: Should `quoteContent` be added to the data model, or is `[回复: quoteId]` acceptable?
   - Recommendation: Accept `[回复: shortId]` without content for Phase 40.1. Adding `replyContent` to `MessageEventData` is a separate concern (requires `listener.ts` changes and `session.quote.content` access).

3. **`trigger` section rendering**
   - What we know: `horizon-view.mustache` has a `<trigger>` section that also renders message observations using `{{{.}}}`
   - What's unclear: Should `trigger` items also use `{{>history-item}}` partial?
   - Recommendation: Yes — `trigger` items are message observations and should render with the same format. Both `history` and `trigger` arrays become `HistoryItemData[]`.
