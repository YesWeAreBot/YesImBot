# Phase 16.2: Percept Type Cleanup & Session Decoupling - Research

**Researched:** 2026-02-21
**Domain:** TypeScript type architecture, session decoupling, tool activation guards
**Confidence:** HIGH

## Summary

Phase 16.2 addresses three interrelated problems left over from 16.1/quick-1: (1) the `BasePerceptRef` temporary type in `shared/types.ts` that creates redundancy with the real `Percept` in `agent/types.ts`, (2) `session` being embedded in `PerceptInput.runtime` which couples the Percept concept to message-based events, and (3) no mechanism to conditionally disable tools based on runtime context (e.g., no session available).

The codebase is small and well-structured. Session is currently used in exactly 5 places across `core/src`: loop.ts fallback send (line 127), loop.ts FunctionContext construction (line 44), horizon service buildView self-info resolution (line 119-123), horizon service environment name resolution (line 146-153), and plugin tools send_message/session-info. The refactor is surgical: replace session threading through Percept with a parallel `ToolExecutionContext`, make `buildView` accept `Scope` instead of `PerceptInput`, and add activator-based tool filtering.

**Primary recommendation:** Introduce `ToolExecutionContext` as a parallel construct to `Percept` (both built from `HorizonMessageEvent` in agent), redesign `Percept` as a clean semantic type in `shared/types.ts`, change `buildView(scope, options)` to decouple horizon from Percept entirely, and port the activator pattern from the dev reference.

## Architecture Patterns

### Pattern 1: Clean Percept Type (shared/types.ts)

**What:** Replace `BasePerceptRef`, `PerceptInput`, and the agent-local `Percept`/`BasePercept`/`UserMessagePercept` hierarchy with a single flat `Percept` type in `shared/types.ts`.

**Current state (3 overlapping types):**
- `shared/types.ts`: `BasePerceptRef { id, type, scope, timestamp }` + `PerceptInput extends BasePerceptRef { runtime?: { session } }`
- `agent/types.ts`: `BasePercept<T> { id, type, scope, priority, timestamp }` + `UserMessagePercept extends BasePercept { payload, triggerType, runtime? }`

**Target state (1 type):**
```typescript
// shared/types.ts
export type TriggerType = "mention" | "reply" | "keyword" | "random" | "direct" | "timer" | "internal";

export interface Percept {
  id: string;
  type: TriggerType;
  scope: Scope;
  timestamp: Date;
  metadata?: Record<string, unknown>;
}
```

**Why:** The user decision locks `Percept` to these core fields. Message-specific payload (messageId, content, sender, channel) moves to `metadata` or to `ToolExecutionContext`. The `priority` field from `BasePercept` is unused in any decision logic (hardcoded to `5` in `buildPercept`) and can be dropped.

**Impact analysis — what currently reads Percept fields:**

| Consumer | Fields Used | Migration Path |
|----------|-------------|----------------|
| `loop.ts` line 26-29 | `percept.type` (PerceptType check) | Check `percept.type` against TriggerType values instead |
| `loop.ts` line 38 | `userPercept` passed to `buildView()` | Pass `percept.scope` + `viewOptions` instead |
| `loop.ts` line 44 | `userPercept.runtime?.session` | Get from `ToolExecutionContext.session` |
| `loop.ts` line 127 | `userPercept.runtime?.session?.send()` | Get from `ToolExecutionContext.session` |
| `loop.ts` line 137-147 | `userPercept.scope` | Use `percept.scope` directly (unchanged) |
| `service.ts` line 134-153 | `event.*` → `UserMessagePercept` fields | Construct `Percept` (slim) + `ToolExecutionContext` (rich) |
| `service.ts` line 224 | `percept` passed to `buildView()` | Pass `percept.scope` |
| `horizon/service.ts` line 108-124 | `percept.scope`, `percept.runtime?.session` | Accept `Scope` + `ViewOptions { session?, botName? }` |
| `horizon/types.ts` line 129 | `HorizonView.percept: BasePerceptRef` | Remove `percept` field from `HorizonView` |
| `plugin/types.ts` line 23 | `FunctionContext.percept?: Percept` | Remove or keep as optional metadata |

### Pattern 2: ToolExecutionContext (parallel to Percept)

**What:** A runtime context object constructed alongside Percept in the agent layer, carrying session and other execution-time dependencies that tools need.

**Recommended fields:**
```typescript
// plugin/types.ts (or shared/types.ts)
export interface ToolExecutionContext {
  scope: Scope;
  session?: Session;
  bot?: Bot;           // ctx.bots.find() result — for cross-channel sends
  percept?: Percept;   // optional reference for tools that want trigger metadata
}
```

**Why these fields:**
- `scope`: Every tool needs to know the channel context. Currently derived from `percept.scope`.
- `session`: Required by `send_message` (default channel send), `session-info`, and platform-specific tools. Optional because non-message events won't have one.
- `bot`: The `send_message` tool already uses `this.ctx.bots.find()` for cross-channel sends. Having `bot` directly avoids re-lookup. Also needed for `self` info resolution that currently lives in `buildView`.
- `percept`: Optional back-reference. Some tools may want to know trigger type for logging.

**Construction site — agent/service.ts `buildPercept` becomes two functions:**
```typescript
private buildPercept(event: HorizonMessageEvent): Percept {
  return {
    id: Random.id(),
    type: event.triggerType,
    scope: event.scope,
    timestamp: event.timestamp,
    metadata: { messageId: event.payload.messageId, content: event.payload.content,
                senderId: event.payload.senderId, senderName: event.payload.senderName },
  };
}

private buildToolContext(event: HorizonMessageEvent): ToolExecutionContext {
  return {
    scope: event.scope,
    session: event.runtime?.session,
    bot: event.runtime?.session?.bot,
  };
}
```

### Pattern 3: buildView Decoupling (horizon layer)

**What:** Change `buildView(percept: PerceptInput)` to `buildView(scope: Scope, options?: ViewOptions)`.

**Current session usage in buildView:**
1. **self info** (line 119-123): `session?.bot?.selfId` and `session?.bot?.user?.name` — used to identify the bot in the view
2. **environment name** (line 146-153): `session?.event?.channel?.name`, `session?.bot?.getChannel()` — used to resolve channel display name

**ViewOptions recommendation:**
```typescript
export interface ViewOptions {
  session?: Session;   // for environment name resolution (API calls)
  selfId?: string;     // bot's self ID
  selfName?: string;   // bot's display name
}
```

**Why keep session in ViewOptions:** The `getOrCreateEnvironment` method (line 148-153) calls `session.bot.getChannel()` to resolve channel names when not cached. This is an API call that genuinely needs a session/bot reference. Rather than duplicating the bot API surface, passing session as an optional in ViewOptions is pragmatic. The key difference: session is now an **optional optimization** for environment resolution, not a structural dependency.

**HorizonView type change:** Remove `percept: BasePerceptRef` field. The view is pure data — environment, entities, history. The percept reference was only used for passing scope through, which is now explicit.

### Pattern 4: Activator-Based Tool Guards

**What:** Declarative activator functions on tool/action definitions that filter tools from the LLM tool list at runtime.

**Dev reference pattern** (verified from `references/YesImBot-dev/packages/core/src/services/plugin/activators.ts`):
```typescript
// Activator signature in dev version
export type Activator = (ctx: GuardContext) => Promise<ActivatorResult>;
export interface ActivatorResult { allow: boolean; reason?: string[] }
```

**Recommended adaptation for v4 (synchronous, simpler):**
```typescript
// plugin/types.ts
export type ActivatorFn = (ctx: ToolExecutionContext) => boolean;

export interface Activator {
  check: ActivatorFn;
  reason?: string;
  onFail?: "remove" | "hint";  // default: "remove"
}

// Preset activators — plugin/activators.ts
export function requireSession(reason?: string): Activator {
  return { check: (ctx) => !!ctx.session, reason: reason ?? "Requires active session", onFail: "remove" };
}

export function requirePlatform(platform: string | string[], reason?: string): Activator {
  const platforms = Array.isArray(platform) ? platform : [platform];
  return {
    check: (ctx) => !!ctx.session?.platform && platforms.includes(ctx.session.platform),
    reason: reason ?? `Requires platform: ${platforms.join(" or ")}`,
    onFail: "remove",
  };
}
```

**Why synchronous:** The dev version uses `async` activators, but all current activators (`requireSession`, `requirePlatform`) are synchronous checks. Async adds complexity for no current benefit. If async is needed later, the interface can be extended.

**Integration into FunctionDefinition:**
```typescript
export interface FunctionDefinition {
  name: string;
  description: string;
  type: FunctionType;
  parameters: Schema;
  activators?: Activator[];  // NEW
  handler: (params: Record<string, unknown>, ctx: ToolExecutionContext) => Promise<ToolResult>;
}
```

**Filtering in PluginService.getTools():**
The current `getTools()` is synchronous and returns all tools unconditionally. It needs to accept a `ToolExecutionContext` and filter:
```typescript
getTools(execCtx?: ToolExecutionContext): Array<{ ... }> {
  // for each tool, check activators against execCtx
  // onFail: 'remove' → skip tool entirely
  // onFail: 'hint' → include tool but append "(unavailable: reason)" to description
}
```

**Decorator integration:**
The `@Tool` and `@Action` decorators need an `activators` field in their options:
```typescript
@Action({
  name: "send_message",
  description: "Send a message",
  parameters: withInnerThoughts({ content: Schema.string().required() }),
  activators: [requireSession()],
})
```

### Anti-Patterns to Avoid

- **Passing session through Percept "just in case":** The whole point is decoupling. If a new consumer needs session, it should get it from ToolExecutionContext.
- **Making HorizonView aware of Percept:** The view is a data snapshot. Percept is a decision artifact. They serve different purposes.
- **Async activators without need:** Keep activators synchronous until a real async use case emerges.
- **Putting ToolExecutionContext in shared/types.ts:** It depends on `Session` from koishi and is consumed by the plugin layer. It belongs in `plugin/types.ts`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Tool filtering | Custom per-tool if/else in loop.ts | Activator pattern on FunctionDefinition | Declarative, composable, matches dev reference |
| Bot lookup for sends | Re-lookup `ctx.bots.find()` in every tool | `ToolExecutionContext.bot` | Single lookup at construction time |

## Common Pitfalls

### Pitfall 1: Breaking the Fallback Send Path
**What goes wrong:** `loop.ts` line 127 uses `userPercept.runtime?.session?.send()` as a fallback when no `send_message` tool was called. After removing session from Percept, this breaks silently (undefined?.send() is a no-op).
**How to avoid:** The loop must receive `ToolExecutionContext` alongside `Percept` and use `toolCtx.session?.send()` for the fallback path. This is the most critical migration point.

### Pitfall 2: Self-Info Resolution in buildView
**What goes wrong:** `buildView` currently gets `selfId` and `selfName` from `percept.runtime?.session?.bot`. After decoupling, if `ViewOptions` doesn't carry this info, the bot identity in the view becomes empty strings.
**How to avoid:** Agent must pass `selfId`/`selfName` (or session) via `ViewOptions` when calling `buildView`. The agent has access to the session from `HorizonMessageEvent.runtime` and can extract bot info before constructing the Percept.

### Pitfall 3: FunctionContext → ToolExecutionContext Migration
**What goes wrong:** The current `FunctionContext` interface is `{ session?, view?, percept?, [key: string]: unknown }`. Changing this to `ToolExecutionContext` affects all tool handler signatures. If the index signature `[key: string]: unknown` is removed, existing tools that pass extra data break.
**How to avoid:** `ToolExecutionContext` should either extend or replace `FunctionContext`. Since the user decision says tools get info from `ToolExecutionContext`, the cleanest path is to replace `FunctionContext` with `ToolExecutionContext` and keep the index signature for extensibility.

### Pitfall 4: HorizonView.percept Removal Breaks Template Scope
**What goes wrong:** The prompt service passes `{ view }` as initial scope to `renderToString`. If any injection or template references `view.percept`, it breaks.
**How to avoid:** Check all injection `renderFn` callbacks. Currently only `__default_identity` reads `view.self` and `view.environment` — neither reads `view.percept`. Safe to remove.

### Pitfall 5: Deferred Judgment Uses buildView with Percept
**What goes wrong:** `agent/service.ts` line 224 calls `horizon.buildView(percept)` in `executeDeferredJudgment`. This must also be migrated to `buildView(scope, options)`.
**How to avoid:** The deferred judgment path must construct ViewOptions from the stored percept's scope and the original event's session (if still valid). Since deferred judgment fires after a delay, the session reference may be stale — this is an existing risk, not new.

## Open Questions

1. **ToolExecutionContext placement**
   - What we know: It depends on `Session` (koishi) and `Scope` (shared). Plugin layer consumes it.
   - What's unclear: Should it live in `plugin/types.ts` or `shared/types.ts`?
   - Recommendation: `plugin/types.ts` — it's a plugin-layer concern. Shared types should stay framework-agnostic where possible. The `Percept` type (no Session dependency) stays in shared.

2. **botName resolution for ViewOptions**
   - What we know: `HorizonService.config.botName` is the current override source. Agent doesn't have direct access to horizon config.
   - What's unclear: Should agent pass `botName` from its own config, or should horizon keep using its own config internally?
   - Recommendation: Keep `botName` resolution inside `HorizonService.buildView`. ViewOptions provides `selfId` and `session` (for API lookups). Horizon applies its own `config.botName` override internally. This avoids config duplication.

3. **Deferred judgment session staleness**
   - What we know: Deferred judgment stores a percept and fires after a delay. The session from the original event may be stale.
   - What's unclear: Is this a real problem in practice?
   - Recommendation: Out of scope for 16.2. The deferred path should store `ToolExecutionContext` alongside percept, accepting that session may be stale. This matches current behavior.
