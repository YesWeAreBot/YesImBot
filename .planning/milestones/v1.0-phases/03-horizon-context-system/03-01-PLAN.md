  <name>Task 1: Create Horizon type definitions</name>
  <files>plugins/core/src/services/horizon/types.ts</files>
  <action>
Create types.ts with all Horizon type definitions, porting from dev version (`YesImBot-dev/packages/core/src/services/horizon/types.ts`) with these v4 simplifications:

**Scope** — same as dev: `{ platform?, channelId?, guildId?, isDirect? }`. No userId/scopeId fields.

**Timeline types:**

- `TimelineEventType` enum: keep `Message = 'message'`, add `AgentSummary = 'agent.summary'`. Remove all dev-only types (Command, MemberJoin, MemberLeave, StateUpdate, Reaction, AgentThought, AgentTool, AgentAction, ToolResult) per user decision "Event types v4 only cover: message and bot self-message (with tool call summary)".
- `TimelinePriority` enum: keep as-is from dev (Noise=0, Normal=1, Important=2, Core=3).
- `TimelineStage` enum: keep as-is from dev (New, Active, Archived, Deleted).
- `BaseTimelineEntry<Type, Data>` interface: same as dev (id, timestamp, scope, type, priority, stage, data).
- `MessageEventData`: same as dev (messageId, senderId, senderName, content, replyTo?).
- `MessageRecord`: `BaseTimelineEntry<TimelineEventType.Message, MessageEventData>`.
- `AgentSummaryData`: new type `{ summary: string }` — compressed text of a full think-act cycle.
- `AgentSummaryRecord`: `BaseTimelineEntry<TimelineEventType.AgentSummary, AgentSummaryData>`.
- `TimelineEntry`: union of `MessageRecord | AgentSummaryRecord`.

**Entity types:**

- `EntityRecord` for DB storage: `{ id, type, name, parentId?, refId?, attributes: Record<string, any>, updatedAt: Date }`. No avatar or createdAt (simpler than dev).
- `Entity` runtime interface: `{ id, type, name, attributes? }`.
- `Environment` interface: `{ type, id, name, description?, metadata: Record<string, any> }`. Same as dev.
- `SelfInfo`: `{ id, name }`.

**Observation types:**

- `MessageObservation`: `{ type: 'message', timestamp, sender: Entity, messageId, content }`.
- `AgentSummaryObservation`: `{ type: 'agent.summary', timestamp, summary: string }`.
- `Observation`: union of the two.

**Percept types:**

- `TriggerType`: `'mention' | 'reply' | 'keyword' | 'random' | 'direct'` — v4 addition not in dev.
- `PerceptType` enum: keep `UserMessage = 'user.message'` only. Remove SystemSignal and TimerTick (not needed in v4).
- `BasePercept<T>`: `{ id, type: T, scope: Scope, priority, timestamp }`.
- `UserMessagePercept` extends BasePercept with `payload` (messageId, content, sender, channel) + `triggerType: TriggerType` + `runtime?: { session: Session }`.
- `Percept`: just `UserMessagePercept` for now.

**HorizonView:**

- `{ percept, self: SelfInfo, environment?, entities?: Entity[], history?: Observation[] }`. Remove mode, workingHistory, memories, index signature from dev version.

**EventQueryOptions** interface: `{ scope?, types?: TimelineEventType[], limit?, since?: Date, until?: Date, orderBy?: 'asc' | 'desc' }`. Export this for EventManager consumers.

Do NOT include Koishi declaration merging here — that goes in service.ts (Plan 03).

<task type="auto">
  <name>Task 2: Create EventManager for Timeline persistence</name>
  <files>plugins/core/src/services/horizon/event-manager.ts</files>
  <action>
Create event-manager.ts porting from dev version (`YesImBot-dev/packages/core/src/services/horizon/event-manager.ts`) with v4 simplifications.

**EventManager class:**

- Constructor takes `ctx: Context` only (no config — simpler than dev).
- Create logger once in constructor: `const logger = ctx.logger('horizon')` per CLAUDE.md.

**Methods:**

1. `record(entry: TimelineEntry): Promise<TimelineEntry>` — wraps `ctx.database.create('yesimbot.timeline', entry)`.

2. `query(options: EventQueryOptions): Promise<TimelineEntry[]>` — port from dev:
   - Build query object with scope filter, optional type `$in` filter, optional timestamp `$gte`/`$lte` range.
   - Chain `ctx.database.select('yesimbot.timeline').where(query)`.
   - Apply `.orderBy('timestamp', options.orderBy)` if set.
   - Apply `.limit(options.limit)` if set.
   - Return `.execute()`.

3. `recordMessage(data: { scope, stage, timestamp, data: MessageEventData }): Promise<MessageRecord>` — convenience method:
   - Generates `id` via `Random.id()` from koishi.
   - Sets `type: TimelineEventType.Message`, `priority: TimelinePriority.Normal`.
   - Calls `this.record(...)`.

4. `recordAgentSummary(data: { scope, timestamp, summary: string }): Promise<AgentSummaryRecord>` — convenience method:
   - Generates `id` via `Random.id()`.
   - Sets `type: TimelineEventType.AgentSummary`, `priority: TimelinePriority.Normal`, `stage: TimelineStage.Active`.
   - Calls `this.record(...)`.

5. `toObservations(entries: TimelineEntry[], selfId?: string): Observation[]` — v4 simplified format:
   - For `Message` entries: create `MessageObservation` with sender Entity from data.
   - For `AgentSummary` entries: create `AgentSummaryObservation` with summary text.
   - Return array in same order as input.

6. `markAsActive(scope: Scope, before?: Date): Promise<void>` — port from dev: updates stage from New to Active for entries matching scope/timestamp.

Import `Random` from 'koishi' for ID generation. Import all needed types from './types'.

Do NOT use `@ts-expect-error` — use proper typing. If Koishi's Query types don't match, use `as any` with a comment explaining why.
</action>
