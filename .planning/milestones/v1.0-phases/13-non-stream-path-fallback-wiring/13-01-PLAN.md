  <action>
**errors.ts**: Add `status === 503` check returning `ErrorCategory.TRANSIENT` — place it alongside the existing 429 check.

**service.ts** — refactor ModelService with these changes:

1. **Private `resolveModel` helper**: Extract the duplicated model-string-or-ModelSelector parsing from call()/streamCall()/getModel() into a single private method `resolveModel(model: string | ModelSelector): { provider: string; modelId: string }` that throws on invalid input.

2. **Private `withRetry` helper**: Add a generic retry wrapper:
   ```ts
   private async withRetry<T>(fn: () => Promise<T>, retries = 1): Promise<T>
   ```
   - Loop `retries + 1` times. On catch, call `classifyError()`. If NOT TRANSIENT and NOT RATE_LIMIT, rethrow immediately. If retries remain, log warn `Retry ${i+1}/${retries} for transient error` and continue. After all retries exhausted, rethrow last error.

3. **Update `call()` signature**: Add optional third parameter `fallbackModel?: string | ModelSelector`. Inside the queue callback:
   - Wrap `executeCall` with `withRetry(fn, 1)`.
   - On transient/rate-limit failure after retries: if `fallbackModel` provided, log warn about switching, try `executeCall` with resolved fallbackModel. If that also fails, fall through to `handleFallback` (global chain). If no fallbackModel, go directly to `handleFallback`.
   - Warn log format: `Primary model failed, trying fallback: ${fallbackId}`

4. **Update `streamCall()` signature**: Same optional `fallbackModel` parameter. Apply identical retry + per-call-fallback + global-chain pattern. Use `resolveModel` for parsing. Wrap the streamText call in `withRetry`.

5. **Update `handleFallback` and `handleStreamFallback`**: Add warn log `Trying fallback chain model: ${fallback}` inside the loop before each attempt.

Keep `getModel()` public (other consumers may use it) but refactor it to use `resolveModel` internally.
