# Phase 16.1: Percept Ownership & User Message Context Refactor - Context

**Gathered:** 2026-02-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Percept 构造从 horizon 移到 agent 模块，horizon 只负责数据记录和事件广播；user message 承载全部动态工作负载（环境、成员、历史、触发），system prompt 变为纯静态；统一使用模板路径渲染上下文。

</domain>

<decisions>
## Implementation Decisions

### Horizon 事件广播设计
- 每条消息独立广播一个事件，horizon 完全无状态（不做聚合、不做分类）
- 事件名使用 `horizon/event`，未来可分流为更具体的事件名（如 horizon/reaction 等）
- 事件携带提取的 payload + 可选的 runtime（koishi Session 引用），为未来非 koishi 事件留空间
- horizon 先记录到 timeline，再广播事件，确保 agent 收到事件时 buildView() 能看到最新消息
- 事件类型使用事件映射表 + 声明合并模式（扩展 koishi Events 接口），与 koishi 自身模式一致
- 本 phase 只定义 message 事件类型，其他事件类型留给后续 phase

### User message 上下文组织
- 排列顺序：环境 → 成员 → 历史 → 触发（触发在最后，最受 LLM 关注）
- 沿用现有 XML 标签格式分隔各 section，未来改格式只改模板
- 缺失数据的 section 保留标签 + 占位文本（LLM 能感知该维度存在但无数据）
- trigger info 作为模板内的 section（非独立 user message），所有上下文在一条 user message 中完整呈现
- Timeline 中 `New` stage 的消息归入 trigger section，history 只包含 `Active` 消息，避免同一条消息重复出现

### 聚合窗口迁移策略
- 迁移 + 增强：逻辑从 listener 搬到 agent，同时改进
- 窗口时长可配置（不再硬编码 1500ms）
- 每条 horizon/event 独立计算意愿，窗口只控制"多久不响应算一轮结束"
- 窗口结束后以最后一条事件为主触发构造 Percept（替代当前的高优先级覆盖逻辑）

### 迁移顺序与兼容性
- Big bang 一步到位，不保留中间兼容状态
- 实现顺序：正常响应路径优先，deferred judgment 最后适配
- 旧上下文渲染路径（toStructured + envLines 手工拼接）全部删除
- Percept 类型定义从 horizon/types.ts 移到 agent 模块

### Claude's Discretion
- 事件映射表的具体接口设计细节
- horizon-view.mustache 模板的具体调整
- 占位文本的措辞
- deferred judgment 适配的具体实现方式

</decisions>

<specifics>
## Specific Ideas

- 事件类型系统参照 koishi Events 的声明合并模式，保持风格一致
- "最后一条事件为主触发"替代"高优先级覆盖"——语义更符合对话直觉（回应最新的消息）
- Timeline 的 `New`/`Active` stage 区分天然适合 trigger vs history 的分流

</specifics>
