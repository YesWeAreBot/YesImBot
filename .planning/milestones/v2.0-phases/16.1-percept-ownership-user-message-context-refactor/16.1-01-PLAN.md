<objective>
Refactor the horizon→agent data flow boundary: horizon broadcasts raw events, agent owns Percept construction and aggregation.

Purpose: Establish correct ownership — horizon is a data layer (record + broadcast), agent is the decision layer (willingness + aggregation + Percept construction). This is the prerequisite for the rendering pipeline changes in Plan 02.

Output: Working event flow where listener emits `horizon/message`, agent subscribes, computes willingness per-event, manages configurable aggregation window, and constructs Percept from last event.
</objective>

<task type="auto">
  <name>Task 2: Agent takes over Percept construction and aggregation window</name>

  <action>
**agent/service.ts:**
- Add Percept type definitions at top of file (moved from horizon/types.ts):
  - `PerceptType` enum (just `UserMessage = "user.message"`)
  - `BasePercept<T>` interface with `id, type, scope, priority, timestamp`
  - `UserMessagePercept` extending `BasePercept` with `payload`, `triggerType`, `runtime`
  - `Percept` union type
- Update imports: import `HorizonMessageEvent`, `TriggerType`, `Scope` from horizon/types (no longer import `Percept`/`UserMessagePercept` from there)
- Import `Random` from koishi (needed for Percept id generation)
- Add `aggregationWindow` field to `AgentCoreConfig` interface (number, optional) and add to `AgentCoreConfigSchema` with `Schema.number().default(1500)`
- Add private field: `pendingWindows = new Map<string, { cancel: () => void; lastEvent: HorizonMessageEvent }>()`
- Change `start()`: replace `this.ctx.on("horizon/percept", ...)` with `this.ctx.on("horizon/message", (event) => this.handleEvent(event))`
- Add `handleEvent(event: HorizonMessageEvent)` method:
  1. Compute `channelKey` from `event.scope`
  2. Call `this.cancelDeferred(channelKey)` (existing)
  3. Compute willingness via `this.willingness.processMessage(channelKey, event.triggerType, event.payload.content)` — same as current but using event fields
  4. Log willingness result (same format as current)
  5. If `!result.shouldReply`: check deferred judgment threshold, schedule if applicable, return
  6. If `event.scope.isDirect`: construct Percept immediately via `this.buildPercept(event)`, enqueue, return
  7. Otherwise (group): reset aggregation window — cancel existing window for channelKey, set new timer via `this.ctx.setTimeout()` with `this.config.aggregationWindow ?? 1500`. On expiry: delete from pendingWindows, construct Percept from stored lastEvent, enqueue.
  8. Store `{ cancel, lastEvent: event }` in `pendingWindows`
- Add `buildPercept(event: HorizonMessageEvent): UserMessagePercept` method:
  ```
  return {
    id: Random.id(),
    type: PerceptType.UserMessage,
    scope: event.scope,
    priority: 5,
    timestamp: event.timestamp,
    triggerType: event.triggerType,
    payload: {
      messageId: event.payload.messageId,
      content: event.payload.content,
      sender: { id: event.payload.senderId, name: event.payload.senderName },
      channel: { id: event.scope.channelId ?? "", platform: event.scope.platform ?? "", guildId: event.scope.guildId },
    },
    runtime: event.runtime,
  };
  ```
- Remove `handlePercept()` method (replaced by `handleEvent()`)
- Update `gateAndEnqueue()`: this method is now replaced by the logic inside `handleEvent()`. Remove it.
- Update `executeDeferredJudgment()`: it still receives a `Percept` — this works because agent now constructs Percept locally. No change needed to its internal logic.
- Update loop.ts import: change `import type { Percept, UserMessagePercept } from "../horizon/types"` to import from `"./service"` (or re-export from agent module). Also update `import { PerceptType } from "../horizon/types"` similarly.
  </action>
