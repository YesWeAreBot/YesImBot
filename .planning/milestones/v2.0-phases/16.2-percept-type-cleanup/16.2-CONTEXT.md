# Phase 16.2: Percept Type Cleanup & Session Decoupling - Context

**Gathered:** 2026-02-21
**Status:** Ready for planning

<domain>
## Phase Boundary

彻底清理 16.1 遗留的类型问题：移除临时方案 BasePerceptRef，将 session 从 PerceptInput 核心结构中解耦，建立工具可用性守卫机制，明确 Percept 在 agent/horizon/plugin 三层的清晰职责边界。

</domain>

<decisions>
## Implementation Decisions

### Session 解耦策略
- session 完全不进入 Percept 体系，从 PerceptInput 中移除 `runtime?: { session: Session }`
- 采用**执行上下文注入**方案：agent 在启动 loop 时构造独立的 ToolExecutionContext（含 session/bot/ctx），工具从执行上下文获取 session 而非从 Percept
- 非消息事件源（定时器、内部触发）构造的 Percept 没有 session 时，ToolExecutionContext 中 session 为 undefined，守卫自动降级禁用需要 session 的工具

### BasePerceptRef 统一方案
- 删除 BasePerceptRef 和 PerceptInput，重新定义一个干净的 Percept 类型
- Percept 核心字段：`id, type (TriggerType), scope, timestamp, metadata?: Record<string, unknown>`
- 类型定义放在 `shared/types.ts`，作为跨模块共享类型
- TriggerType 扩展以支持非消息事件源（如 timer、internal 等）

### 工具可用性守卫
- 采用**声明式 activators** 模式，类似 dev 版的 activators 数组
- 守卫粒度为**工具方法级**，每个工具方法单独声明自己的 activators
- 设计通用 Activator 接口（如 `(ctx: ToolExecutionContext) => boolean`），requireSession 和 requirePlatform 为预置实现
- Activator 可声明失败策略 `onFail: 'remove' | 'hint'`，默认 `remove`（静默从 LLM 工具列表移除），`hint` 模式保留工具但标记不可用并附带原因

### Percept 三层职责边界
- **horizon 层**：纯数据层，完全不知道 Percept 类型的存在。buildView() 签名改为 `buildView(scope: Scope, options?: ViewOptions)`，不再接收 Percept
- **agent 层**：Percept 的唯一构造者。从 HorizonMessageEvent 提取信息构造 Percept，同时平行构造 ToolExecutionContext，两者一起传给 loop
- **plugin 层**：工具只通过 ToolExecutionContext 获取所需信息（scope, session 等），不直接接触 Percept 对象

### Claude's Discretion
- ToolExecutionContext 的具体字段设计
- ViewOptions 的具体字段
- TriggerType 新增的非消息事件源枚举值命名
- Activator 接口的具体签名细节

</decisions>

<specifics>
## Specific Ideas

- 平台特定接口（如 session.onebot）只存在于 session 上，这是保留 session 传递路径的核心原因——不能完全绕过 session
- buildView() 内部当前只用 percept 来取 scope 和传递 session，session 解耦后可以直接接收 Scope 参数
- Percept 保留为独立类型，代表"一次响应决策"的语义单元——告诉 LLM 为什么被唤醒、用于响应追踪和 agent 内部调度
- 参考 dev 版 `requirePlatform('onebot', 'OneBot platform required')` 和 `requireSession('Active session required')` 的 activator 模式

</specifics>
