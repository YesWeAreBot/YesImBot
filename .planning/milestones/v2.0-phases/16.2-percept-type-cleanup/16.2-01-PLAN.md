**horizon/service.ts** — Modify `buildView` and `getOrCreateEnvironment`:

   - Import `ViewOptions` from `"./types"` (add to existing import)

2. Change `buildView` signature from `buildView(percept: PerceptInput)` to `buildView(scope: Scope, options?: ViewOptions)`:
   ```ts
   async buildView(scope: Scope, options?: ViewOptions): Promise<HorizonView> {
     const { platform, channelId } = scope;
     // ... query entries same as before ...
     const history = this.events.toObservations(entries);
     const environment = await this.getOrCreateEnvironment(scope, options?.session);
     const entities = await this.getEntities(scope);
     const self = {
       id: options?.selfId ?? "",
       name: this.config.botName || options?.selfName || options?.selfId || "",
     };
     return { self, environment: environment ?? undefined, entities, history };
   }
   ```
   Key changes:
   - `percept.scope` → `scope` (direct parameter)
   - `percept.runtime?.session` → `options?.session` for environment resolution
   - Self info from `options?.selfId` / `options?.selfName` instead of `session?.bot`
   - `botName` override stays in horizon (uses `this.config.botName` — per research recommendation)
   - Remove `percept` from the returned HorizonView object

3. Change `getOrCreateEnvironment` signature from `(scope, runtime?)` to `(scope, session?)`:
   ```ts
   private async getOrCreateEnvironment(scope: Scope, session?: Session): Promise<Environment | null> {
   ```
   - Replace `runtime?.session` references with direct `session` parameter
   - `const session = runtime?.session` line is removed — session is now the parameter directly
   - `session?.event?.channel?.name`, `session?.bot?.getChannel()` calls remain unchanged (they use session directly)
   - Add `Session` to the import from koishi if not already there

4. Remove `percept` from the `return` statement in `buildView` — the `HorizonView` type no longer has that field.
