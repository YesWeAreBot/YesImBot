# Phase 16.4: working-memory-improve - Context

**Gathered:** 2026-02-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Agent 循环（ThinkActLoop）内的工作记忆动态裁剪：替代当前 messages 数组线性增长无裁剪的现状，实现渐进式工具调用结果裁剪，并将完整响应记录到 timeline。持久化记忆检索、embedding 索引等不在此 phase 范围内。

</domain>

<decisions>
## Implementation Decisions

### 裁剪策略
- 每轮开始前检查 messages 总量是否超过阈值，超过则触发裁剪
- 两级渐进降级：softTrim（保留头尾固定字符数，中间省略）→ hardClear（替换为占位符，仅保留工具名）
- 保留最近 N 轮完整工具结果，N 可配置，默认 2
- 更早的轮次先 softTrim，再早的 hardClear

### Timeline 记录（替代 Summary）
- 移除现有 AgentSummary 事件类型，替换为 AgentResponse 类型
- AgentResponse 存完整数据：assistant 原始文本 + 解析后的工具调用 + 工具结果
- 每轮实时写入 timeline（一轮一条事件），不等循环结束
- 裁剪只影响内存中的 messages 数组，不影响 timeline 中的完整记录

### Token 预算分配
- 固定字符数上限（不按模型窗口比例），可配置
- 用字符数估算，不引入 tokenizer 依赖
- 工具结果和 assistant 回复都可被裁剪
- 裁剪优先级：先裁工具结果，不够再裁 assistant 回复

### 上下文呈现方式（分层）
- history 中按时间顺序包含所有事件（群聊消息 + agent 行为摘要），保持因果关系
- agent 行为在 history 中只显示摘要行（工具名列表），完整工具调用详情放在独立的 working-memory section
- send_message 特殊处理：在 history 摘要行中展示完整发送内容（等同于 agent 发言），working-memory 中也保留（允许重复）
- 数据源与模板分离：裁剪逻辑提供数据源，具体格式由 mustache 模板定义
- 提示词中需说明 history（对话视角）和 working-memory（执行视角）两种视角的关系，指导 LLM 如何利用上下文

### Claude's Discretion
- softTrim 的具体头尾字符数默认值
- 固定字符数上限的默认值
- AgentResponse 的具体数据结构设计
- working-memory section 的模板格式

</decisions>

<specifics>
## Specific Ideas

- 参考 OpenClaw 的 contextPruning 模式：softTrim（headChars + tailChars）→ hardClear（placeholder）两级裁剪
- agent 通过工具发送消息，且 agent 发的消息不以 message 事件记录到 timeline，因此工作记忆是 agent 自身经历事件的唯一来源
- 两种视角的设计理念：history 是"对话流"（谁说了什么、agent 做了什么），working-memory 是"执行日志"（工具调用的完整参数和结果）

</specifics>
