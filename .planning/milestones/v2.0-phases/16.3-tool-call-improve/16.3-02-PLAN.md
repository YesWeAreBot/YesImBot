**Rewrite `loop.ts`** — replace the entire ThinkActLoop.run() method:

1. **Remove** all AI SDK tool-related imports (`LanguageModelUsage`, `StepResult`, `ToolSet`). Remove `LoopAbort` class. Remove `buildAiSdkTools`, `buildStopCondition` imports.

2. **Add imports** for `JsonParser` from `./json-parser`, `buildToolSchemaForPrompt` from `./tools`, `FunctionType` from plugin types.

3. **Define AgentResponse interface** in loop.ts (or a local types section):
```typescript
interface AgentResponse {
  thoughts?: { observe: string; analyze_infer: string; plan: string };
  actions: Array<{ name: string; params?: Record<string, unknown> }>;
  request_heartbeat?: boolean;
}
```

4. **Define ToolResultEntry interface:**
```typescript
interface ToolResultEntry {
  id: number;
  name: string;
  status: string;
  result?: unknown;
  error?: string;
}
```

5. **Rewrite `run()` method** with manual heartbeat loop (see RESEARCH.md Pattern 1):
   - Build system prompt via `prompt.renderToString("system", ...)` — same as current
   - Build user content via `horizon.formatHorizonText(view)` — same as current
   - Inject tool schema into prompt: call `buildToolSchemaForPrompt(pluginService, toolCtxWithPercept)` and register it as a `basic_functions` injection via `prompt.inject()` BEFORE rendering. Dispose after loop completes.
   - Initialize `messages` array with initial user message
   - Manual loop: `while (round < maxRounds)`
     - Call `modelService.call()` with `{ system: systemPrompt, messages }` — NO tools, NO toolChoice, NO stopWhen, NO onStepFinish
     - Parse `result.text` using `JsonParser<AgentResponse>`
     - If parse fails: attempt LLM repair if raw text contains "actions" substring (simple secondary call with repair prompt using same model). If repair also fails, break loop.
     - Execute actions: partition into Tool-type and Action-type using `pluginService.getDefinition(name)?.type`. Execute Tool-type in parallel (`Promise.allSettled`), Action-type sequentially. Collect `ToolResultEntry[]`.
     - Determine continuation: `autoShouldContinue = hasToolCalls && !hasActionCalls`. `shouldContinue = parsed.request_heartbeat ?? autoShouldContinue`.
     - If `!shouldContinue` → break
     - If `round >= maxRounds && shouldContinue` → force wrap-up: append assistant msg + user msg with wrap-up prompt, do one final call, break
     - Otherwise: append `{ role: "assistant", content: result.text }` + `{ role: "user", content: formatToolResults(toolResults) }` and continue
   - After loop: record agent summary (same as current). No fallback text sending — per locked decision, send_message is the only reply channel.

6. **Add helper functions** in loop.ts:
   - `formatToolResults(results: ToolResultEntry[]): string` — compact JSON format: `"Tool results:\n" + JSON.stringify(compactResults)`
   - `formatFinalRoundPrompt(results: ToolResultEntry[]): string` — wrap-up prompt telling model to call send_message with final response

7. **Update `service.ts`** — add `enableThoughts` to `AgentCoreConfig`:
   - Add `enableThoughts?: boolean` field with `Schema.boolean().default(true)` in config schema
   - This config is passed to the loop but the loop itself doesn't enforce it — it's used by the template (thoughts field is optional in AgentResponse)

**Key constraints from locked decisions:**
- actions is ALWAYS an array, even for single tool call
- Tool results use auto-assigned IDs (array index), model does not generate IDs
- Tool-type parallel, Action-type sequential
- No content direct reply — only send_message
- JSON parse failure: jsonrepair → LLM repair (if "actions" present in raw text) → abandon
