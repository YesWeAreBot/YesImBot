# Phase 16.3: Tool Call Improve - Context

**Gathered:** 2026-02-22
**Status:** Ready for planning

<domain>
## Phase Boundary

重新设计 tool call 实现方式：从原生 tool_call 切换到结构化文本（JSON）输出，恢复对交互流的完全控制权。解决原生 tool_call 的模型兼容性问题、AI SDK 耦合问题、循环控制问题。

</domain>

<decisions>
## Implementation Decisions

### 调用范式
- 放弃原生 tool_call，切换到纯结构化文本（JSON）输出
- 模型输出 JSON 对象包含 actions 数组，代码解析并执行
- AI SDK 降级为纯传输层：不传 tools 参数，只负责发请求和流式处理
- 工具 schema 在提示词中描述，不依赖 AI SDK 的 tools 参数

### 循环控制机制
- 按工具类型自动推断是否继续循环 + 模型可通过 request_heartbeat 字段覆盖
  - Tool 类型（获取信息）→ 自动继续循环（结果需回流）
  - Action 类型（执行操作）→ 自动结束循环
  - request_heartbeat: true/false 可覆盖默认行为
- 移除 finish 工具，不再需要显式终止信号
- 最大轮次可配置（不同场景可调整），达到上限后强制收尾回合（再请求一次模型生成最终回复）

### 上下文组装策略
- 工具结果嵌入 user message（与 16.1 架构一致：system prompt 纯静态）
- 多轮 heartbeat 使用追加消息对：首轮完整 user message，后续追加 assistant（模型 JSON 输出）+ user（工具结果）
- 工具结果使用紧凑 JSON 格式，减少 token 消耗
- actions 始终为数组格式，即使只有一个工具调用
- Tool 类型并行执行，Action 类型顺序执行
- 工具结果按自动分配的 ID 关联（模型不生成 ID，后台按数组索引分配）

### 回复质量保障
- 纯工具回复路径：send_message 是唯一与用户交流的渠道，不允许 content 直接回复
- JSON 解析失败处理：强健 JSON 解析器 → 必要字段存在时 LLM 修复 → 无法修复则放弃响应
- thoughts 字段为可选配置：用户可选择是否启用，有推理能力的模型可不需要

### Claude's Discretion
- JSON schema 的具体字段设计（在 v3 基础上精简）
- 提示词中工具 schema 的呈现格式
- 强健 JSON 解析器的具体实现（可参考 v3/dev 已有实现）
- 强制收尾回合的提示词设计
- 可配置上限的默认值

</decisions>
