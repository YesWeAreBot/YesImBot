# Phase 16.3: Tool Call Improve - Research

**Researched:** 2026-02-22
**Domain:** Agent loop architecture — structured text JSON output replacing native AI SDK tool_call
**Confidence:** HIGH

## Summary

This phase replaces the current native AI SDK `tool_call` mechanism (via `generateText`/`streamText` with `tools` parameter) with structured JSON text output parsed and executed by application code. The v3 codebase already validated this exact pattern with `HeartbeatProcessor` + `JsonParser` + `StreamParser`, and the v4 codebase retains the `FunctionType.Tool` vs `FunctionType.Action` distinction needed for automatic loop continuation inference.

The core change is in `ThinkActLoop` (`core/src/services/agent/loop.ts`) and `tools.ts` (`core/src/services/agent/tools.ts`). Currently, `loop.ts` passes `tools` and `toolChoice: "required"` to `ModelService.call()` which delegates to AI SDK's `generateText()`. The new approach: do NOT pass `tools` to AI SDK at all — instead describe tools in the system prompt, parse the model's JSON text output, execute tools manually, and append results as user messages for subsequent rounds.

**Primary recommendation:** Rewrite `ThinkActLoop` to use a manual heartbeat loop with JSON parsing, reuse v3's `JsonParser` (with `jsonrepair`) and `StreamParser` patterns, and move tool schema rendering into the `basic_functions` prompt injection point.

## Architecture Patterns

### Current Architecture (to be replaced)
```
loop.ts
├── buildAiSdkTools() → converts FunctionDefinitions to AI SDK ToolSet
├── ModelService.call() with { tools, toolChoice: "required", stopWhen }
├── AI SDK internally manages tool call → result → next step loop
└── onStepFinish callback + LoopAbort for early termination

tools.ts
├── finishTool — explicit termination signal
├── buildAiSdkTools() — wraps PluginService functions as AI SDK tools
└── buildStopCondition() — hasToolCall("finish") | stepCountIs(maxRounds)
```

### New Architecture
```
loop.ts (rewritten)
├── Build system prompt (static, includes tool schemas via prompt injection)
├── Build initial user message (horizon view)
├── Manual heartbeat loop:
│   ├── Call ModelService (no tools param, pure text generation)
│   ├── Parse JSON response (JsonParser with jsonrepair)
│   ├── Execute actions (Tool parallel, Action sequential)
│   ├── Determine continuation (auto-infer from types + request_heartbeat override)
│   ├── If continuing: append assistant msg (raw JSON) + user msg (tool results)
│   └── If max rounds reached: one final "wrap up" round
└── Fallback: if no send_message called, discard (pure tool path)

tools.ts (simplified)
├── buildToolSchemaForPrompt() — renders tool definitions for prompt injection
└── Remove: finishTool, buildAiSdkTools, buildStopCondition
```

### Pattern 1: Manual Heartbeat Loop
**What:** Application-controlled loop replacing AI SDK's internal multi-step tool calling
**When to use:** Always — this is the new default
**Example:**
```typescript
// Pseudocode for the new loop structure
const messages: CoreMessage[] = [{ role: "user", content: userContent }];
let round = 0;

while (round < maxRounds) {
  round++;
  const result = await modelService.call(model, {
    system: systemPrompt,
    messages,
    // NO tools param — AI SDK is pure transport
  }, fallbackChain);

  const parsed = parseAgentResponse(result.text);
  if (!parsed) break; // JSON parse failed after all recovery attempts

  const { toolResults, hasToolCalls, hasActionCalls } = await executeActions(parsed.actions);

  // Auto-infer continuation
  const autoShouldContinue = hasToolCalls && !hasActionCalls;
  const shouldContinue = parsed.request_heartbeat ?? autoShouldContinue;

  if (!shouldContinue || round >= maxRounds) {
    if (round >= maxRounds && shouldContinue) {
      // Force wrap-up round
      messages.push({ role: "assistant", content: result.text });
      messages.push({ role: "user", content: formatFinalRoundPrompt(toolResults) });
      await modelService.call(model, { system: systemPrompt, messages }, fallbackChain);
    }
    break;
  }

  // Append for next round
  messages.push({ role: "assistant", content: result.text });
  messages.push({ role: "user", content: formatToolResults(toolResults) });
}
```

### Pattern 2: Tool Schema in Prompt
**What:** Tool definitions rendered as structured text in system prompt instead of AI SDK tools parameter
**When to use:** Always — replaces AI SDK tool schema passing
**Example:**
```typescript
// Tool schema rendered into the basic_functions injection point
// Format: compact JSON-like description in system prompt
function buildToolSchemaText(pluginService: PluginService, toolCtx: ToolExecutionContext): string {
  const entries = pluginService.getTools(toolCtx);
  return entries.map(e => {
    const typeLabel = e.functionType === FunctionType.Tool ? "tool" : "action";
    return `- ${e.function.name} (${typeLabel}): ${e.function.description}\n  Parameters: ${JSON.stringify(e.function.parameters)}`;
  }).join("\n");
}
```

### Pattern 3: Continuation Inference
**What:** Automatic loop continuation based on function types, with model override
**When to use:** After each round of action execution
**Logic:**
- If ANY action in the round is `FunctionType.Tool` → auto-continue (results need to flow back)
- If ALL actions are `FunctionType.Action` → auto-stop (side effects executed, done)
- `request_heartbeat` field in model output overrides the auto-inference
- Mixed Tool+Action in same round: Tool results flow back, Actions already executed

### Anti-Patterns to Avoid
- **Passing tools to AI SDK:** The entire point is to NOT use AI SDK's tool calling. Never pass `tools` or `toolChoice` to `generateText`/`streamText`.
- **Using AI SDK message types for tool results:** Tool results go in user messages as plain text/JSON, not as `role: "tool"` messages.
- **Parsing JSON without repair:** Always use `jsonrepair` before giving up. LLM output is frequently malformed (trailing commas, unclosed strings, markdown wrapping).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON repair | Custom regex-based fixer | `jsonrepair` library | Handles 20+ edge cases (truncated output, trailing commas, unquoted keys, comments, etc.) |
| Streaming JSON parse | New streaming parser | Port v3's `StreamParser` class | Already battle-tested with the exact same schema structure |
| Tool schema serialization | Manual string building | Reuse `schemaToJSONSchema()` from `plugin/schema.ts` | Already converts Koishi Schema to JSON Schema objects |

**Key insight:** The v3 codebase (`references/YesImBot-v3/`) already implemented and validated this exact pattern. The `JsonParser`, `StreamParser`, and `HeartbeatProcessor` are proven implementations that should be ported, not reinvented.

## Common Pitfalls

### Pitfall 1: JSON Wrapped in Markdown Code Blocks
**What goes wrong:** Models frequently wrap JSON output in ` ```json ... ``` ` blocks
**Why it happens:** Training data includes markdown-formatted code
**How to avoid:** The v3 `JsonParser` already handles this — it strips markdown code blocks before parsing. Port this logic.
**Warning signs:** Parse failures on otherwise valid JSON

### Pitfall 2: Forgetting to Handle Mixed Tool+Action Rounds
**What goes wrong:** Model calls both a Tool (get_weather) and an Action (send_message) in the same round
**Why it happens:** Model decides to fetch info and act in one step
**How to avoid:** Execute Tools first (parallel), then Actions (sequential). For continuation: if any Tool was called, results need to flow back even though Actions already executed. The `request_heartbeat` override handles edge cases.
**Warning signs:** Tool results never reaching the model after mixed rounds

### Pitfall 3: Infinite Loop on Persistent request_heartbeat
**What goes wrong:** Model always sets `request_heartbeat: true`, loop never terminates
**Why it happens:** Model doesn't understand when to stop
**How to avoid:** Hard max rounds limit (already in config as `maxRounds`). Force wrap-up round at limit.
**Warning signs:** Token costs spiking, response latency growing

### Pitfall 4: AI SDK generateText Still Expecting Tool Results
**What goes wrong:** If `tools` param is accidentally passed, AI SDK enters its internal tool-calling loop
**Why it happens:** Leftover code from current implementation
**How to avoid:** Completely remove `tools`, `toolChoice`, `stopWhen`, and `onStepFinish` from the call params. The `CallParams` type from AI SDK still allows these — must not use them.
**Warning signs:** AI SDK making multiple internal requests per call

### Pitfall 5: System Prompt Becoming Dynamic
**What goes wrong:** Tool schemas change per-request (different tools available based on context), making system prompt non-cacheable
**Why it happens:** Tool availability depends on `ToolExecutionContext` (activators filter tools)
**How to avoid:** Accept that `basic_functions` injection point content varies per-request. The `CACHEABLE_POINTS` set in `prompt/service.ts` already excludes non-static points. Ensure `basic_functions` is NOT in the cacheable set (it currently IS — this needs to change).
**Warning signs:** Stale tool lists being sent to the model

## State of the Art

| Old Approach (current v4) | New Approach (this phase) | Impact |
|---------------------------|---------------------------|--------|
| AI SDK native tool_call with `tools` param | Structured JSON text output, manual parsing | Full control over execution flow, model-agnostic |
| AI SDK internal multi-step loop | Manual heartbeat loop in application code | Control over context assembly, token efficiency |
| `finish` tool for termination | Auto-inference from FunctionType + request_heartbeat | Simpler model instructions, one fewer tool |
| `toolChoice: "required"` forcing tool calls | JSON output format instruction in prompt | More natural model output, better response quality |
| Tool results as AI SDK `role: "tool"` messages | Tool results in `role: "user"` messages as JSON text | Consistent with 16.1 architecture (system=static, user=dynamic) |

**Deprecated/outdated:**
- `buildAiSdkTools()` in `tools.ts` — entire function replaced
- `finishTool` in `tools.ts` — removed, no longer needed
- `buildStopCondition()` in `tools.ts` — removed, manual loop control
- `LoopAbort` class in `loop.ts` — removed, clean loop exit instead
- AI SDK imports `hasToolCall`, `stepCountIs`, `jsonSchema` — no longer needed

## Open Questions

1. **Stream mode support**
   - What we know: v3 had `StreamParser` for streaming JSON parsing. Current v4 has `streamMode` config option.
   - What's unclear: Whether streaming JSON parse is needed in this phase or can be deferred. The v3 `StreamParser` is complex (~250 lines).
   - Recommendation: Implement non-streaming first, add streaming as a follow-up if needed. The `StreamParser` from v3 can be ported later. Keep `streamMode` config but initially only support non-streaming in the new loop.

2. **LLM-based JSON repair fallback**
   - What we know: CONTEXT.md mentions "必要字段存在时 LLM 修复" as a fallback after jsonrepair fails.
   - What's unclear: Exact trigger conditions and which model to use for repair.
   - Recommendation: Implement as a simple secondary call with a repair prompt when jsonrepair fails but the raw text contains recognizable field names (e.g., "actions" substring present). Use the same model. Keep it simple — if repair also fails, abandon the response.

3. **CACHEABLE_POINTS for basic_functions**
   - What we know: `basic_functions` is currently in `CACHEABLE_POINTS` in `prompt/service.ts`. Tool schemas are dynamic (depend on activators/context).
   - What's unclear: Whether this was intentional or an oversight.
   - Recommendation: Remove `basic_functions` from `CACHEABLE_POINTS` since tool availability varies per request.
